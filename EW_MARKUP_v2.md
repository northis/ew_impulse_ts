# Алгоритм разметки волн Эллиотта — версия 2 (проектный документ)

Этот документ описывает **вторую версию** движка автоматической разметки волн
Эллиотта. Версия 1 (`ElliottWaveExactMarkup`) остаётся без изменений; v2
реализуется как **отдельный параллельный движок** `ElliottWaveExactMarkupV2`.

Правила моделей и фибо-соотношения **переиспользуются** из v1 без изменений:
- `ElliottWavePatternHelper.ModelRules` — допустимые под-модели в каждой позиции.
- Фибо-карты из `ElliottWaveExactMarkupFibo.cs` (см. `EW_RULES.md`).
- `NotationHelper` — 15 уровней нотации (`level % 15`).

Связанные документы:
- `EW_MARKUP.md` — алгоритм v1 (рекурсивный перебор сегментов).
- `EW_RULES.md` — каталог моделей, фибо-карты, правила генерации.
- `EW_PREDICTION.md` — режим прогноза незавершённых моделей (база для §13 ниже).

> **Статус:** проектный документ. Код v2 реализован, числовые пороги частично
> откалиброваны (см. §16.3). Инструмент пошаговой отладки — `TradeKit.ReplayViewer`
> (§13.1, §17.1).

## 1. Мотивация и отличия от v1

### 1.1 Недостатки v1

| # | Проблема v1 | Причина |
|---|---|---|
| P1 | Не каждый участок размечается | Перебор `TryCombinations` требует `S ≥ K` сегментов целиком; на неудобных диапазонах модель не собирается |
| P2 | Разметка зависит от указанного входного участка | Старт/конец задаются извне (глобальный max/min последних N баров), смещение границ → другая разметка |
| P3 | Проблемы ранжирования точек начала/конца волн | Граничные точки подволн «чинятся» постфактум (`RepairSimpleImpulseLeaves`, `SyncLeafBoundaries`), что порождает рассинхрон и хрупкие правки |
| P4 | Высокая сложность | `IMPULSE` даёт `O(n^4)` комбинаций; спасают только жёсткие кэпы (`MAX_ITERATIONS_PER_MODEL = 500 000`) |
| P5 | Нет непрерывности | Разметка одного диапазона не стыкуется с соседним |

### 1.2 Ключевые идеи v2

1. **Снизу вверх (bottom-up), а не сверху вниз.** Вместо «возьми диапазон и подбери
   к нему модель целиком» мы идём слева направо по сегментам зигзага минимального
   периода и наращиваем дерево гипотез о том, частью какой модели является каждый
   сегмент.
2. **Точки границ — это вход, а не результат.** Зигзаг минимального периода
   гарантирует строгое чередование min/max и совпадение экстремумов с границами
   сегментов. Алгоритм **не чинит** границы — он на них опирается (устраняет P3).
3. **Инкрементальная отбраковка (pruning).** Каждый узел дерева имеет условия
   валидности по цене и по времени. Невалидный узел отмирает вместе со всеми
   потомками — это режет дерево на ранних уровнях (бьёт по P4).
4. **Непрерывность как цель.** Алгоритм должен разметить **всю историю** из файлов
   `data/` без «дыр» (устраняет P5, проверяется тестами — см. §15).
5. **Прогноз вместо обрезки.** Конец диапазона обычно приходится на текущий бар;
   незавершённые модели не отбрасываются, а проецируются (§13).

## 2. Терминология и инварианты входных данных

| Термин | Значение |
|---|---|
| **Сегмент** | Отрезок зигзага между двумя соседними экстремумами. Минимальная единица анализа. Тип по умолчанию — `SIMPLE_IMPULSE`. |
| **Пивот (pivot)** | Экстремум зигзага (`BarPoint`): цена + индекс бара + время. |
| **Узел (node)** | Гипотеза «диапазон сегментов `[i..j]` = модель `M` в позиции `pos`». |
| **Кандидат** | Полное поддерево разметки от корня до листьев на текущий момент. |
| **Ранг / уровень** | Глубина вложенности модели (см. §12, `NotationHelper`). |
| **Активный сегмент** | Последний сегмент, упирающийся в текущий бар; волна может быть не завершена (см. §13). |

### 2.1 Инварианты входного зигзага (гарантируются построителем)

- **I1. Строгое чередование.** Пивоты строго чередуются: min, max, min, max, …
  Алгоритм опирается на это и **не** проверяет повторно.
- **I2. Экстремум = граница.** Для каждого сегмента его экстремальная цена
  (high для восходящего, low для нисходящего) совпадает с одной из границ
  сегмента. Внутри сегмента нет бара, пробивающего его границы (свойство
  `SIMPLE_IMPULSE`, §10).
- **I3. Минимальная длительность.** Каждый сегмент ≥ 2 баров (старт ≠ конец).
- **I4. Полнота.** Зигзаг покрывает весь анализируемый диапазон без разрывов.

> Если построитель зигзага нарушает I1–I3 (а в v1 такие случаи всплывали в тестах),
> это **дефект построителя**, а не движка разметки. v2 чинить вход не должна —
> вместо этого добавляется тест-инвариант на сам зигзаг.

## 4. Входной зигзаг минимального периода

- На вход подаётся зигзаг **минимального периода** — ниже по масштабу не спускаемся.
  Все сегменты этого зигзага — атомарные `SIMPLE_IMPULSE`.
- **Период (девиация) задаётся пользователем.** Значение по умолчанию — результат
  `DeviationOptimizer.FindOptimalDeviation()`: точка насыщения (saturation point) —
  наименьшая девиация, при которой дальнейшее её уменьшение уже не увеличивает
  число сегментов зигзага. Построитель по умолчанию — `SimpleExtremumFinder`
  с этой девиацией (тот же подход, что в `IterativeElliottWaveExactIndicator`).
- **Принцип точки насыщения:** дальнейшее уменьшение девиации уже не рождает
  новых экстремумов — значит мы достигли «дна» разрешения для данного диапазона.
  Это и есть «минимальный период». Пользователь может переопределить девиацию
  явно (например, ради производительности на большой истории).

> **Уточнение к v1:** в v1 финер-экстремумы доискивались внутри подволн
> (`SUBWAVE_REDISCOVERY_DEVIATION`) уже в процессе разметки. В v2 весь зигзаг
> строится **один раз заранее**, и движок к нему не добавляет точек — это убирает
> источник рассинхронизации границ (P3).

## 5. Каталог моделей и стартовый набор

### 5.1 Источник истины

Допустимые под-модели в каждой позиции волны берутся из
`ElliottWavePatternHelper.ModelRules[parentModel].Models[waveKey]` — **тот же
словарь, что и в v1**. Дублировать правила в v2 нельзя; только переиспользовать.

### 5.2 Стартовый набор моделей

Первый сегмент зигзага — это импульсное движение, значит он может быть только
**началом** модели, начинающейся с импульса. Стартовый набор корневых моделей:

| Модель | Роль первого сегмента |
|---|---|
| `IMPULSE` | волна 1 |
| `DIAGONAL_CONTRACTING_INITIAL` | волна 1 |
| `ZIGZAG` | волна a |
| `DOUBLE_ZIGZAG` | волна w → a |

> Все эти модели начинаются с под-модели из множества «импульсных»
> (`IMPULSE / DIAGONAL_CONTRACTING_INITIAL`). То есть первый сегмент трактуется
> как **волна 1** или **волна a**. Коррективные модели, начинающиеся с коррекции,
> корнем быть не могут.
>
> **Расширяющиеся (EXPANDING) диагонали** сами по себе очень редки и в стартовый
> набор **не входят** (на текущем этапе). Они допустимы только как под-модели в
> соответствующих позициях согласно `ModelRules`. Аналогично `COMBINATION`,
> `TRIPLE_ZIGZAG` и `FLAT_REGULAR` корнем не стартуют — только как под-модели.

### 5.3 Замечание о направлении

Направление первого сегмента (`IsUp`) задаёт ориентацию всего корневого паттерна.
Противоположное направление будет рассмотрено как отдельный корень, если первый
сегмент окажется, например, концом предыдущей структуры (стыковка диапазонов, §15).

## 6. Дерево вариантов: узел и его жизненный цикл

### 6.1 Состав узла

```
TreeNode {
    id:            уникальный идентификатор (для экспорта §17)
    parentId:      идентификатор родителя (null для корня)
    model:         ElliottModelType
    wavePos:       ключ позиции ("1".."5", "a".."e", "w"/"x"/"y"/...)
    level:         уровень вложенности (§12)
    startPivot:    индекс пивота-начала
    endPivot:      индекс пивота-конца (или -1, если волна ещё формируется)
    isUp:          направление
    status:        ALIVE | DEAD | COMPLETE | PROJECTED
    deathReason:   null | enum (см. §6.4)
    score:         накопленный скоринг (§16)
    children:      TreeNode[]   // дочерние гипотезы (под-волны / расширения)
}
```

### 6.2 Жизненный цикл

1. **Рождение.** Узел создаётся как гипотеза «следующий сегмент(ы) — это волна
   `wavePos` модели `model`».
2. **Рост.** По мере прихода новых сегментов узел либо принимает их как свои
   под-волны, либо порождает дочерние узлы-гипотезы под-моделей.
3. **Проверка.** После каждого присоединённого сегмента проверяются условия
   отмирания (§7–9). Нарушение hard-условия → `status = DEAD`.
4. **Смерть.** Мёртвый узел делает мёртвыми **всех потомков** (см. §6.3). Братья
   при этом могут оставаться живыми.
5. **Завершение.** Когда все ожидаемые волны набраны и валидны → `COMPLETE`.
6. **Проекция.** Если волн не хватает, но диапазон упёрся в текущий бар →
   `PROJECTED` (§13).

### 6.3 Каскад смерти, но не каскад братьев

Это ключевая оптимизация. Пример из ТЗ (п.6):

> Импульс, и в текущем узле предполагается, что волна 4 = `SIMPLE_IMPULSE`, и она
> пробила уровень волны 1. Узел «4 = SIMPLE_IMPULSE» **умирает**. Но его брат
> «4 = TRIANGLE_CONTRACTING / FLAT_RUNNING» **остаётся живым**: треугольник/бегущий
> флет в волне 4 может закончиться вне зоны волны 1, и импульс уцелеет.

Следствие: условие отмирания **привязано к конкретной гипотезе модели в узле**,
а не к диапазону цен вообще. Один и тот же сегментный диапазон порождает несколько
узлов-братьев с разными моделями и разными условиями жизни.

### 6.4 Причины смерти (`deathReason`)

| Код | Описание | Раздел |
|---|---|---|
| `PRICE_BREACH` | Цена пробила запрещённый уровень для данной модели/позиции | §7 |
| `TIME_WINDOW` | Конец волны вне допустимого временного окна | §8 |
| `DURATION_ORDER` | Младшая волна длительнее старшей (нарушение 90%-правила) | §9 |
| `SIMPLE_OVERSHOOT` | `SIMPLE_IMPULSE` имеет вылет за свои границы (нарушение I2) | §10 |
| `EXTENSION_CANCELLED` | Сработал уровень/время отмены растяжения | §11 |
| `NO_VALID_SUBMODEL` | Не нашлось ни одной валидной под-модели для позиции | §6.5 |
| `RATIO_OUT_OF_RANGE` | Фибо-соотношение вне допустимого диапазона (hard-вариант) | §16 |

### 6.5 Бэктрекинг по уровням (п.8 ТЗ)

Если умер **последний** живой узел на уровне, проверяется его родитель; если у
родителя не осталось живых детей — родитель тоже мёртв, и проверка поднимается
выше, пока не найдётся уровень хотя бы с одним живым вариантом. Реализуется как
обновление счётчика живых детей у родителя при смерти ребёнка (без рекурсивного
обхода всего дерева).

## 7. Условия отмирания: цена (hard-price)

Ценовые hard-правила берутся из классических правил Эллиотта и совпадают с v1.
Проверяются **инкрементально**, при присоединении каждого сегмента.

### 7.1 IMPULSE

| Правило | Условие смерти |
|---|---|
| Волна 2 не уходит за начало волны 1 | конец W2 за `start(W1)` |
| Волна 3 не самая короткая | `len(W3) < len(W1)` **и** `len(W3) < len(W5)` |
| Волна 4 не заходит в зону волны 1 (для `SIMPLE_IMPULSE` в W4) | перекрытие W4↔W1 (см. оговорку §6.3 для треугольника/флета) |
| Волна 3 всегда импульс | под-модель W3 ∉ импульсных |
| **Волна 5 пробивает конец волны 3** | конец W5 внутри зоны W3 (усечение) |

> **Усечения (truncations):** согласно Фросту и Пректеру, усечения волны 5
> допускаются в теории Эллиотта. Однако в настоящей версии (v2) усечения
> **исключены** — считается, что конец волны 5 обязан выйти за конец волны 3.
> Нарушение этого условия приводит к отмиранию узла (`PRICE_BREACH`). Это
> сделано потому, что усечения крайне редки на практике и их допущение
> порождает значительное число ложноположительных разметок.

### 7.2 ZIGZAG / коррекции

| Правило | Условие смерти |
|---|---|
| ZIGZAG: волна B не уходит за начало A | конец B за `start(A)` (с допуском) |
| ZIGZAG: волна C пробивает конец A **и** экстремум B | C не дошла до конца A, либо не пробила экстремум B (если B имела внутренние волны, выходящие за конец A) |
| FLAT_*: волна B **обязана** выйти за начало A | B не достигла начала A |
| FLAT_RUNNING: итоговая коррекция ≥ 38.2 % от волны A | `s · (Start(A) − End(C)) < 0.382 · |A|` (знаковое: C на стороне коррекции) |
| FLAT_EXTENDED: волна C пробивает конец волны A | C не дошла до конца A |
| FLAT_REGULAR: B не уходит сильно за origin | вылет B за origin сверх допуска |
| TRIANGLE_CONTRACTING: каждая волна короче предыдущей | волна длиннее предыдущей |
| TRIANGLE_running: B превышает начало треугольника | B не превысила |

### 7.3 Допуск (tolerance)

Hard-правила (§7.1–§7.2) проверяются **без допуска** (сравнение строгое,
`tol ≈ 0`): цена либо пересекла границу, либо нет. Никакого «почти касания»
для структурных запретов.

Для **soft-правил** (фибо-соотношения, §16) действует асимметричный принцип:

> **Фибо-уровень засчитывается, только если он достигнут или пробит.**
> «Почти касание» (цена не дошла до уровня) — **ноль** бонуса за этот уровень.
>
> Награда за попадание максимальна при точном совпадении с уровнем ($\text{ratio} = r_i$)
> и экспоненциально затухает с ростом превышения:
> $$\text{score}_i = w_i \cdot \exp\left(-k \cdot \frac{\text{ratio} - r_i}{r_i}\right)$$
>
> Между двумя фибо-зонами награда определяется по **ближайшему достигнутому** (нижнему)
> уровню и убывает вплоть до следующей зоны. Для последней (максимальной) фибо-зоны
> затухание продолжается до зоны отмены модели.

Таким образом, `MAIN_ALLOWANCE_MAX_RATIO` (5%) из v1 для hard-правил **не используется**
в v2; для soft-правил его роль выполняет сам механизм экспоненциального затухания —
«почти касание» получает околонулевой скоринговый бонус, а не проходит фильтр.

## 8. Условия отмирания: длительность (hard-time)

Новизна v2 относительно v1: **явные временные окна** для ещё не наблюдаемых волн.

### 8.1 Принцип

Если набраны волны 1–3 импульса, то волна 4 не может закончиться раньше или позже
временного окна, согласованного с длительностью волны 2 (волны одного ранга
сопоставимы по времени). Узел-гипотеза, чей конец волны выпадает из окна,
отмирает с `TIME_WINDOW`.

### 8.2 Формализация окна

Для волны `Wk`, у которой есть «эталон того же ранга» `Wref` (например, W4↔W2,
W5↔W1, X↔W в зигзагах):

```
durRef          = bars(Wref)
[durMin, durMax] = [k_min · durRef, k_max · durRef]
```

Конец `Wk` допустим, только если его длительность ∈ `[durMin, durMax]`.
Коэффициенты `k_min`, `k_max` — **TBD**, калибруются по тестам (§16). Ориентир:
`k_min ≈ 0.3`, `k_max ≈ 3.0` (широкий диапазон на старте, сужается по данным).

### 8.3 Эталоны по моделям (черновик, уточняется тестами)

| Модель | Пара эталонов того же ранга |
|---|---|
| IMPULSE | W2 ↔ W4; W1 ↔ W3 ↔ W5 (мягко) |
| ZIGZAG | A ↔ C (по времени сопоставимы) |
| DOUBLE_ZIGZAG | W ↔ Y; X между ними |
| TRIANGLE_* | соседние волны убывают и по цене, и по времени (мягко) |

### 8.4 Коррекции дольше импульсов (статистическое временное ожидание)

Ключевое временное ожидание (мягкое, влияет на `score`, но может быть переведено
в hard по тестам):

> **Коррекции статистически длиннее по времени, чем импульсные волны.** Отсюда:
> импульс в волнах **1 / a** ожидается **короче по времени**, чем последующая
> коррекция в волнах **2 / b**. Если наблюдается обратное (импульс 1/a длиннее
> коррекции 2/b) — кандидат получает штраф (или отмирает, если порог
> превышен сильно).
>
> **Исключение — диагонали.** Если импульсная позиция (1 / a / c / 5) выполнена
> диагональю (`DIAGONAL_*`), её длительность может быть **сопоставима** с
> коррекцией — для диагоналей этот временной штраф не применяется.

### 8.5 Геометрия диагоналей и треугольников (образующие)

Для диагоналей и треугольников временные окна имеют **меньший вес**; главный
критерий — **геометрический контур** подволн.

**Когда строятся образующие.** Образующие линии (trendlines) можно построить
только тогда, когда уже собраны **концы подволн текущего ранга**, задающие эти
линии:
- **Треугольник:** образующие появляются, когда известны концы волн **A–D**
  (линия по A-C и линия по B-D). Финальная волна E проверяется на касание /
  непробитие сходящихся образующих.
- **Диагональ:** образующие появляются, когда известны концы волн **1–4**
  (линия по концам 1-3 и линия по концам 2-4). Волна 5 проверяется на
  попадание в коридор образующих.

> **До** этого момента (пока волны A–D / 1–4 ещё не собраны) образующих нет, и
> модель оценивается **только по прочим правилам** для треугольников/диагоналей
> (price-hard §7, убывание/чередование длин, фибо-соотношения §16). Образующие —
> это **поздняя** геометрическая проверка, а не ранний фильтр.

Когда образующие построены:

> Подволны должны **хорошо ложиться в образующие** и **не сильно выходить** за
> них. Образующие строятся **по концам подволн текущего ранга** (а не по точкам
> верификационного ранга). Количественно:
>
> - Для треугольника: вершины A-C(-E) лежат на одной образующей, впадины B-D —
>   на другой; обе сходятся. Вылет конца подволны за свою образующую больше
>   допуска — мягкий штраф пропорционально амплитуде (аналог `overshootPenalty`
>   из v1).
> - Для диагонали: образующие сходятся (contracting) или расходятся (expanding);
>   концы волн 1–3–5 и 2–4 должны лежать близ своих образующих.
>
> Таким образом, для этих моделей **геометрия важнее временных пропорций**
> (см. также light-валидацию флетов/треугольников в `EW_MARKUP.md` §4.6).

> Временные окна на старте — **мягкие** (штраф), и только проверенные тестами
> переводятся в hard. Это снижает риск отбраковки валидных, но «растянутых во
> времени» структур.

## 9. Правило соотношения длительностей старших и младших волн

(п.10 ТЗ — отдельное hard-правило кросс-уровневой гармонии.)

> **Правило DURATION_ORDER:** волна младшего уровня не может быть длительнее
> соответствующей волны старшего уровня. Старшая волна должна составлять **не
> менее 90 %** длительности младшей (порог 0.9, чтобы захватить случаи примерного
> равенства).

Формально, для под-волны `c` на уровне `L+1` внутри родительской волны `P` на
уровне `L`:

```
bars(parentWaveOfSameRole) ≥ 0.9 · bars(childWave)   // иначе DEAD: DURATION_ORDER
```

Применяется в первую очередь к:
- **волне 2 в растяжениях** (волна 2 of 5 не длительнее волны 2 паттерна);
- **волнам x и b в зигзагах**: волна b of w должна быть **короче по времени**, чем
  волна x (`bars(x) ≥ 0.9 · bars(b_of_w)` иначе разметка невалидна).

> Это перекликается с порогами `HARMONY_*` в v1 (`HARMONY_HARD_RATIO = 3.0`,
> `HARMONY_PESSIMIZE_RATIO = 2.0`), но v2 формулирует правило **направленно**
> (младшая ≤ старшая), а не симметрично. Конкретный порог 0.9 — стартовый,
> уточняется тестами.

## 10. SIMPLE_IMPULSE и инвариант «без вылетов»

- `SIMPLE_IMPULSE` — атомарный сегмент: его экстремум **всегда** совпадает с
  началом или концом (инвариант I2). Вылетов нет.
- Это **гарантируется входным зигзагом**, а не чинится движком. Если внутри
  сегмента находится бар, пробивающий границу, — это нарушение I2, и узел
  `SIMPLE_IMPULSE` умирает с `SIMPLE_OVERSHOOT` (а сегмент должен был быть разбит
  построителем зигзага на более мелкие — это сигнал о неверном периоде/девиации).
- `SIMPLE_IMPULSE` может стоять в **любой** под-волне любой модели (как в v1:
  `ModelRules[SIMPLE_IMPULSE]` ≡ `ModelRules[IMPULSE]` с коэффициентом 0.25).

> **Отличие от v1:** в v1 границы `SIMPLE_IMPULSE` доводились до истинного OHLC
> постфактум (`RepairSimpleImpulseLeaves`, правила §4.1-direction-endpoint /
> §4.2-endpoint-repair в `EW_MARKUP.md`). В v2 этой починки **нет** — границы
> уже корректны по построению. Весь блок repair-логики из v1 в v2 не переносится.

## 11. Растяжения (extensions) и зоны отмены

После сборки базового паттерна узел спрашивает: **возможно ли растяжение?**

### 11.1 Типы растяжений

| Где | Что предполагаем | Уровень/время отмены |
|---|---|---|
| Растяжение волны 5 | волны 1–2 of 5 уже видны, ждём 3–5 of 5 | пробитие начала волны 5; либо волна 2 of 5 длительнее волны 2 (§9) |
| Растяжение волны 3 | внутри W3 ещё один импульс старшего масштаба | пробитие начала W3 в обратную сторону |
| Растяжение волны c (в a-b-c) | второй зигзаг = волны 1–2 растяжения c | аналогично W5 |

### 11.2 Зоны отмены (cancellation)

Каждый узел-растяжение хранит **уровень отмены** (цена) и/или **время отмены**:
- **Ценовая отмена:** цена возвращается за начало растягиваемой волны →
  `EXTENSION_CANCELLED`.
- **Временная отмена:** нарушено правило §9 (под-волна 2 длительнее эталонной
  волны 2) → `DURATION_ORDER`.

### 11.3 Кластерные зоны Фибоначчи

Для проекции конца растяжения используется **пересечение** фибо-целей старшего и
младшего уровней (как описано в `EW_PREDICTION.md` §3.3.1). Если кластер найден —
цель = середина кластера, вес = произведение весов уровней; иначе — цель старшего
уровня с пониженным весом.

## 13. Режим прогноза

Разметка начинается от локального экстремума, но **заканчивается на текущем баре**.
Поэтому последние 1–2 волны часто отсутствуют, и мы выдаём **наиболее вероятное
продолжение**.

Полностью переиспользуется логика `EW_PREDICTION.md`:
- **Уровень завершённости** `C = S / K`: допускаются `C = 1`, `(K−1)/K`, `(K−2)/K`.
- **Активный сегмент** — последний отрезок, упирающийся в текущий бар; он не
  обязан быть полной волной и не фиксирован как конец модели (`status = PROJECTED`).
- **Проекции** строятся по фибо-целям и образующим линиям; рисуются пунктиром
  (подтверждённые волны — сплошными).
- На текущем этапе возвращается **один лучший** вариант (архитектура допускает
  список в будущем).

> В v2 проекция — естественная часть жизненного цикла узла (`status = PROJECTED`),
> а не отдельная подсистема `GetProjections` как в v1.

### 13.1 Обновление по закрытию бара и web-отладка (Replay Viewer)

Обновление идёт строго **по закрытию бара** (внутрибаровые движения
игнорируются, как в `EW_PREDICTION.md` G3). Каждый закрытый бар может:
- продлить активный сегмент (цена идёт в ту же сторону);
- закрыть сегмент и открыть новый пивот (разворот по девиации);
- убить часть узлов (price/time/duration) и оживить ранее отложенные ветви.

**Web-инструмент отладки — `TradeKit.ReplayViewer`**

Для наблюдения за работой алгоритма в реальном времени создано веб-приложение
на ASP.NET Core Kestrel. В отличие от первоначального плана (JSON-файлы →
внешний визуализатор), replay-данные **не пишутся на диск** — сервер
генерирует их на лету и отдаёт прямо в браузер.

```
┌─ TradeKit.ReplayViewer (Kestrel :5000) ───────────────────────┐
│                                                               │
│  POST /api/replay/init                                        │
│    { file, fromDate, toDate }                                 │
│         │                                                     │
│         ▼                                                     │
│  ReplayEngine.Initialize(...):                                │
│    1. ReplayBarsProvider.LoadCandles(csv) → PriceDecimals     │
│    2. ElliottWaveExactMarkupV2(provider, startBar, endBar)    │
│    3. возвращает { symbol, timeframe, priceDecimals,          │
│           startBar, endBar, totalSteps }                      │
│                                                               │
│  POST /api/replay/step  (на каждое нажатие)                   │
│    { deadDepth }                                              │
│         │                                                     │
│         ▼                                                     │
│  ReplayEngine.Step(deadDepth):                                │
│    1. продвигается на 1 сегмент зигзага                      │
│    2. раскрывает новые бары (m_LastRevealedBar+1 .. newBar)  │
│    3. BuildSnapshot(sub, result, deadDepth)                   │
│    4. возвращает { done, frame, snapshot, newCandles }        │
│         │                                                     │
│         ▼                                                     │
│  ┌──────────────────────────────────────────┐                │
│  │  Браузер (TradingView Lightweight Charts)│                │
│  │  ┌────────────────┬────────────────────┐ │                │
│  │  │ Свечной график │ Панель узлов       │ │                │
│  │  │ + разметка     │ (автоскролл вниз)  │ │                │
│  │  │ + волны/проекц │ • COMPLETE / ALIVE │ │                │
│  │  │ + инкремент.   │ • DEAD + причина   │ │                │
│  │  │   бары          │                    │ │                │
│  │  └────────────────┴────────────────────┘ │                │
│  │  ▶ Play / ⏸ Pause   ⏭ Step   3 / 39     │                │

└───────────────────────────────────────────────────────────────┘
```

**Возможности web-интерфейса:**
- **Выбор CSV** из `data/` с автоопределением полного диапазона дат
- **Диапазон задаётся датами** (не индексами баров) — поля ввода `dd.mm.yyyy`
  (текстовые, day-first, без зависимости от локали браузера)
- **Пошаговое проигрывание** — кнопка `⏭ Step`: ровно один сегмент зигзага
  за нажатие; сервер выполняет шаг на лету и возвращает снапшот дерева +
  новые бары
- **Авто-проигрывание** — кнопка `▶ Play` (превращается в `⏸ Pause`):
  последовательный вызов шагов с задержкой 150 мс; Pause доигрывает текущий
  сегмент и останавливается
- **Инкрементальное раскрытие баров** — на каждом шагу на график добавляются
  только бары, составляющие текущий сегмент (не все сразу)
- **Точность цены по Y** — число десятичных разрядов извлекается из CSV
  (поле `PriceDecimals`) и передаётся в UI → формат оси Y и цен в панели
  узлов `.toFixed(priceDecimals)`
- **Не кликабельный прогресс-бар** внизу — показывает позицию в диапазоне
  дат (stepIndex / totalSteps) и дату текущего бара
- **Панель узлов** — дерево узлов из снапшота: цветовая индикация статуса
  (✓ зелёный = `COMPLETE`, ⏳ жёлтый = `PROJECTED`/`ALIVE`,
  ✗ красный = `DEAD`), модель, позиция, скор, ценовой диапазон, причина смерти
- **Автоскролл панели узлов вниз** — новые ветви всегда видны
- **Клик по узлу** → на графике рисуется разметка именно этого узла (подволны +
  зоны отмены + пунктирные проекции для `PROJECTED`)

**Внутреннее устройство:**
- `ReplayBarsProvider` — лёгкая реализация `IBarsProvider`, загружающая CSV
  в память; вычисляет `PriceDecimals` (макс. число разрядов OHLC); предоставляет
  `FindBarIndex(DateTime)` для трансляции дат в индексы
- `ReplayEngine` — оркестратор с состоянием:
  - `Initialize(file, fromDate, toDate)` — грузит свечи → строит зигзаг →
    создаёт `ElliottWaveExactMarkupV2` на диапазоне → запоминает пивоты,
    все свечи, стартовый бар; возвращает метаданные (`/api/replay/init`)
  - `Step(deadDepth)` — продвигает `m_CurrentStep` на 1, вызывает
    `EwMarkupTreeExporter.BuildSnapshot()` для текущего префикса пивотов,
    раскрывает новые бары (`newCandles`); возвращает `ReplayStepResult`
    (`/api/replay/step`)
- `ReplayStepResult` — `{ done, frame, snapshot, newCandles }`:
  - `frame` — `EwReplayFrameDto` с `closeTime`, `barIndex`, `newPivot`
  - `snapshot` — `EwTreeSnapshotDto` (§17): полное дерево узлов на текущий шаг
  - `newCandles` — бары, раскрытые на этом шагу (для инкрементального добавления
    на график)
  - `done` — `true`, когда шаги исчерпаны (stepIndex == totalSteps)
- DTO-классы из §17 (`EwTreeSnapshotDto`, `EwNodeDto`, `EwPivotDto`) — внутренний
  формат данных, передаваемый через HTTP API; сериализация — System.Text.Json
  (camelCase, как в ASP.NET Core minimal API по умолчанию)
- **Важно:** `frame.newPivot` **не содержит время** — время текущего бара
  находится в `frame.closeTime` (дата открытия пивота)

**Запуск:**
```powershell
cd TradeKit.ReplayViewer
dotnet run
# → https://localhost:5000
```

Путь к `data/` настраивается переменной окружения `REPLAY_DATA_DIR` (по умолчанию
`..\..\data` от папки проекта).

> Первоначальный план (§17.1 в первой редакции) предполагал экспорт JSON во
> внешние файлы с последующей загрузкой в визуализатор. От этого подхода
> отказались в пользу прямого веб-сервера: он устраняет промежуточный слой,
> позволяет менять диапазон «на лету» и наблюдать работу алгоритма без
> перезапуска.

## 14. Управление комбинаторной сложностью

Это центральная инженерная задача (п.14 ТЗ). Предлагаемые отраслевые практики:

### 14.1 Branch-and-bound с ранней отбраковкой
Каждый узел проверяет hard-условия (§7–9) **немедленно** при присоединении
сегмента. Мёртвая ветвь не разворачивается. Это основной механизм (см. §6.3).

### 14.2 Beam search (лучевой поиск)
На каждом уровне дерева удерживаем не более `BEAM_WIDTH` лучших по `score` живых
узлов на каждый диапазон. Остальные отсекаются. `BEAM_WIDTH` — TBD (ориентир
v1: `MAX_CANDIDATES_FOR_SUBWAVE_FILL = 20`, `MIN_CANDIDATES_PER_MODEL = 3`).
Гарантируем минимум `MIN_PER_MODEL` лучей на каждый тип модели, чтобы редкие
модели не вытеснялись доминирующим IMPULSE (как в v1).

### 14.3 Динамическое программирование / мемоизация
Гипотеза «диапазон сегментов `[i..j]` = модель `M`» **не зависит** от того, как
размечен путь до неё. Кэшируем результат по ключу `(i, j, M, level)`. Это
превращает экспоненциальный перебор в полиномиальный по числу диапазонов
(классический интервальный DP, как в CYK-парсинге грамматик).

> Аналогия: разметка Эллиотта — это **вероятностный парсинг по контекстно-свободной
> грамматике**, где модели = правила продукции, сегменты = терминалы. Подходят
> алгоритмы chart-parsing (CYK / Earley) с весами.

### 14.4 A* / эвристика приоритета
Очередь узлов упорядочивается по `score + h`, где `h` — допустимая (не
переоценивающая) оценка качества достройки оставшихся волн. Это направляет поиск
к перспективным веткам раньше.

### 14.5 Жёсткие кэпы как страховка
Сохраняем абсолютные лимиты (`MAX_ITERATIONS_PER_LEVEL`, `MAX_NODES_TOTAL`) на
случай патологических входов — как в v1 (`MAX_ITERATIONS_PER_MODEL = 500 000`).

### 14.6 Инкрементальность
Зигзаг растёт слева направо; при добавлении нового сегмента пересчитываются только
затронутые открытые узлы, а не всё дерево. Это критично для режима прогноза (§13),
где обновление идёт по закрытию каждого бара.

## 16. Скоринг и определение вероятностей по тестам

### 16.1 Скоринг

Скоринг узла = переиспользованные фибо-веса v1 (`GetFiboWeight`, геометрическое
среднее по соотношениям — см. `ElliottWaveExactMarkupFibo.cs`) **×** вероятностный
коэффициент модели **×** мягкие штрафы (overshoot, time-window, trendline).

```
score(node) = fiboScore × P(model | position) × Π(softPenalties)
```

> **Усечение (truncation) исключено из мягких штрафов:** невыход волны 5 за
> конец волны 3 теперь является **hard-правилом** (§7.1) и приводит к отмиранию
> узла (`PRICE_BREACH`), а не к понижению скора.
>
> **Фибо-диапазон — hard:** если любое ценовое отношение между волнами модели
> превышает максимальный уровень в фибо-карте более чем на 10 %
> (`FIBO_MAX_OVERSHOOT = 1.1`), или попадает в зону 0 %-го веса (ниже первого
> не-сентинельного уровня карты), модель отбрасывается немедленно — **до**
> вычисления фибо-скора — с причиной `RATIO_OUT_OF_RANGE`. Проверка выполняется
> для полностью собранных моделей (все ожидаемые волны набраны).

### 16.2 Вероятности моделей — методология (п.13 ТЗ)
Стартовые коэффициенты берутся из `ModelRules[*].ProbabilityCoefficient`
(см. `EW_RULES.md` §2). Затем **уточняются по тестам**:

1. Прогнать v2 на всей истории `data/` с равными/исходными коэффициентами.
2. Собрать частотность фактически выбранных моделей по позициям
   (`DeathByReason`, итоговые `COMPLETE`-узлы).
3. Оценить эмпирические `P(model | position)` как нормированные частоты при
   максимизации `Coverage` и минимизации `GapCount`.
4. Внести откалиброванные значения в финальный код v2.

## 17. JSON-схема экспорта дерева

Экспорт дерева вариантов в JSON — внутренний формат данных для визуализации
и отладки. Используется web-инструментом `TradeKit.ReplayViewer` (§13.1, §17.1),
который передаёт эти данные через HTTP API напрямую в браузер (без записи на диск).

Схема снапшота (`EwTreeSnapshotDto`):

```json
{
  "$schema": "ew-markup-tree/v2",
  "symbol": "EURUSD",
  "timeframe": "h1",
  "rangeStartBar": 12000,
  "rangeEndBar": 12940,
  "zigzag": [
    { "barIndex": 12000, "price": 1.0850, "isHigh": false },
    { "barIndex": 12017, "price": 1.0920, "isHigh": true  }
  ],
  "nodes": [
    {
      "id": "n0",
      "parentId": null,
      "model": "IMPULSE",
      "wavePos": "root",
      "level": 0,
      "startPivot": 0,
      "endPivot": 8,
      "isUp": true,
      "status": "COMPLETE",
      "deathReason": null,
      "score": 0.83,
      "children": ["n1", "n2", "n3", "n4", "n5"]
    },
    {
      "id": "n4",
      "parentId": "n0",
      "model": "SIMPLE_IMPULSE",
      "wavePos": "4",
      "level": 1,
      "startPivot": 5,
      "endPivot": 6,
      "isUp": false,
      "status": "DEAD",
      "deathReason": "PRICE_BREACH",
      "score": 0.0,
      "children": []
    }
  ]
}
```

Поля `status` и `deathReason` позволяют визуализатору раскрашивать живые/мёртвые
ветви и показывать причину отбраковки. Экспорт включается флагом (например,
`ParseOptions.ExportTreePath`), по умолчанию выключен (нулевой оверхед в проде).

> Дерево может быть большим; для экспорта применяется тот же beam-фильтр (§14.2),
> либо опция «экспортировать и мёртвые ветви до глубины D» для отладки.

### 17.1 Пошаговое проигрывание (replay) — `TradeKit.ReplayViewer`

Пошаговая отладка реализована как веб-приложение на ASP.NET Core Kestrel
(см. §13.1). Сервер выполняет шаги **на лету** (on-demand): каждый запрос
`/api/replay/step` перепрогоняет v2-разметку на растущем префиксе пивотов
и возвращает полный снапшот дерева на текущий шаг.

**Инициализация — `POST /api/replay/init`:**

```json
// Запрос
{ "file": "EURUSD_h1_...csv", "fromDate": "2024-05-01", "toDate": "2024-05-10" }

// Ответ
{
  "symbol": "EURUSD",
  "timeframe": "h1",
  "priceDecimals": 5,
  "startBar": 12000,
  "endBar": 12940,
  "totalSteps": 39
}
```

**Шаг — `POST /api/replay/step`:**

```json
// Запрос
{ "deadDepth": 1 }

// Ответ (промежуточный шаг)
{
  "done": false,
  "frame": {
    "barIndex": 12017,
    "closeTime": "2024-05-01T10:00:00Z",
    "newPivot": { "barIndex": 12017, "price": 1.09204, "isHigh": true }
  },
  "snapshot": {
    "zigzag": [
      { "barIndex": 12000, "price": 1.08503, "isHigh": false },
      { "barIndex": 12017, "price": 1.09204, "isHigh": true }
    ],
    "nodes": [
      {
        "id": "n0",
        "parentId": null,
        "model": "IMPULSE",
        "wavePos": "root",
        "level": 0,
        "startPivot": 0,
        "endPivot": -1,
        "isUp": true,
        "status": "ALIVE",
        "score": 0.0,
        "children": []
      }
    ]
  },
  "newCandles": [
    { "time": "2024-05-01T10:00:00Z", "open": 1.08650, "high": 1.09204,
      "low": 1.08550, "close": 1.09120, "barIndex": 12017 }
  ]
}

// Ответ (последний шаг)
{ "done": true }
```

**Генерация шагов** — `ReplayEngine.Step(deadDepth)`:
- Инкрементирует `m_CurrentStep` (начинается с 2, т.к. нужно ≥2 пивотов для
  первого сегмента)
- Вызывает `ElliottWaveExactMarkupV2.ParseTiled()` на префиксе
  `pivots[0..m_CurrentStep]` и `EwMarkupTreeExporter.BuildSnapshot()`
- Раскрывает бары `m_LastRevealedBar+1 .. newPivot.BarIndex` как `newCandles`
- Если `m_CurrentStep >= pivots.Count` → возвращает `{ done: true }`

**Особенности реализации:**
- **Не pre-compute:** дерево перестраивается на каждом шагу заново на растущем
  префиксе — это O(n²) по числу пивотов, поэтому рекомендуется использовать
  узкие диапазоны дат (например, 3–5 дней для h1)
- **Стабильные ID узлов** — `StableId(node)` =
  `{Model}|{StartSeg}-{EndSeg}|{WavePos}|L{Level}`.
  Один и тот же логический узел сохраняет ID между шагами, что позволяет
  визуализатору отслеживать его жизненный цикл.
- **Снапшот** (`EwTreeSnapshotDto`, см. §17) содержит полные атрибуты
  всех узлов на текущий шаг (включая мёртвые ветви при `deadDepth > 0`).
- **`frame.closeTime` — источник времени:** `newPivot` (EwPivotDto) не содержит
  время — дата текущего бара находится в `frame.closeTime` (дата открытия пивота).
