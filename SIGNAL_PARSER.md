# Парсинг сигналов из Telegram-каналов

## Обзор

Компонент `ParseSetupFinder` (`TradeKit.Core/Signals/ParseSetupFinder.cs`) читает историю сообщений Telegram-канала из JSON-файла и на каждом новом баре отслеживает вход в сделку, изменение стопа, тейкпрофита, безубыточность и факт закрытия.

---

## Форматы входных файлов

Поддерживаются два варианта JSON:

### 1. Экспорт из Telegram Desktop («Экспорт истории чата»)
```json
{
  "messages": [
    {
      "id": 737,
      "date": "2022-10-18T13:22:00+00:00",
      "text": "GOLD SELL - 1655.5\nSL 1661\nTP 1651",
      "reply_to_msg_id": null
    },
    {
      "id": 738,
      "date": "2022-10-18T13:57:00+00:00",
      "text": "move sl to entry",
      "reply_to_msg_id": 737
    }
  ]
}
```

### 2. Массив сообщений напрямую
```json
[
  { "id": 1, "date": "...", "text": "...", "reply_to_msg_id": null },
  ...
]
```

### Поля `TelegramHistoryMessage`

| Поле JSON       | C#-свойство | Описание                                              |
|-----------------|-------------|-------------------------------------------------------|
| `id`            | `Id`        | Уникальный ID сообщения                               |
| `date`          | `Date`      | Дата/время сообщения (UTC offset, напр. `+00:00`)     |
| `text`          | `Text`      | Текст сообщения (может быть массивом rich-text JSON)  |
| `reply_to_msg_id` | `ReplyId` | ID исходного сообщения, на которое это является ответом |

**Важно:** Поле `date` Newtonsoft.Json десериализует с `DateTimeKind.Local`. При загрузке оно принудительно нормализуется в UTC-тики с `DateTimeKind.Unspecified`, чтобы корректно совпадать с временами баров cTrader (которые тоже `Unspecified`-UTC).

## Поддерживаемые инструменты

Соответствие символа определяется через словарь `SYMBOL_REGEX_MAP`. Сейчас поддерживается:

| Символ  | Regex для распознавания в тексте      |
|---------|---------------------------------------|
| XAUUSD  | `(gold\|xauusd\|one\s*more)`          |

Если символ не найден в словаре — парсер молча игнорирует все сообщения.

## Регулярные выражения

### Для входных сигналов (`ProcessMessage`)

| Константа         | Regex                                               | Что ищет                                    |
|-------------------|-----------------------------------------------------|---------------------------------------------|
| `SIGNAL_REGEX`    | `(buy\|sell)([^\n]*?(?:\n[^\d\n@\-]*)?)[@\-]\s*(\d+(?:[.,]\d{0,5})?)` | Направление и цена входа; разделитель `@` или `-` |
| `TP_REGEX`        | `(tp\|take profit)(.*)\s(\d+(?:[.,]\d{0,5})?)` | Тейкпрофит(ы)                               |
| `SL_REGEX`        | `(SL\|stop\s?loss)\D*(\d{1,9}(?:[.,]\d{0,5})?)` | Стоп-лосс                                   |
| `LIMIT_REGEX`     | `(limit)`                                           | Лимитный ордер (ключевое слово)              |

### Для обновляющих сообщений-ответов

| Константа          | Regex                                                                              | Что ищет                                     |
|--------------------|------------------------------------------------------------------------------------|----------------------------------------------|
| `BREAKEVEN_REGEX`  | `(running in profit\|entry point\|break\s*-?\s*even\|to the entry\|b\.e\|move sl to entry\|sl\s+move\s+to\s+(?:the\s+)?entry)` | Перенос стопа в безубыток |
| `TP_HIT_REGEX`     | `(tp\s*\d*\s*hit\|target\s+hit\|🎯)`                                              | Сработал тейкпрофит (поддержка tp1/tp2 hit и «target hit») |
| `SL_HIT_REGEX`     | `(sl\s*(was\s*)?hit\|stop\s*out)`                                                  | Сработал стоп-лосс                           |
| `CLOSE_REGEX`      | `(close[d\s]\|close$)`                                                             | Ручное закрытие (игнорируется при наличии `\bhold\b`) |
| `ACTIVATED_REGEX`  | `(activated)`                                                                      | Активация лимитного ордера                   |
| `CANCELED_REGEX`   | `(delete\|cancel\|remove)`                                                         | Отмена лимитного ордера                      |
| `EXTRA_REGEX`      | `\b(extra)\b`                                                                      | Дополнительный вход по той же идее           |
| `STRIKETHROUGH_REGEX` | JSON-блок `strikethrough`                                                       | Зачёркнутый текст удаляется перед разбором   |

> Весь текст переводится в нижний регистр перед применением regex. Флаг `RegexOptions.IgnoreCase` не нужен, но всё равно проставлен для надёжности.

## Флаги действий (`SignalAction`)

Результат `ProcessMessage` — битовая маска:

| Флаг           | Значение | Описание                          |
|----------------|----------|-----------------------------------|
| `DEFAULT`      | 0        | Ничего не распознано              |
| `ENTER_BUY`    | 1        | Вход в лонг                       |
| `ENTER_SELL`   | 2        | Вход в шорт                       |
| `SET_TP`       | 4        | Установить тейкпрофит             |
| `SET_SL`       | 8        | Установить стоп-лосс              |
| `CLOSE`        | 16       | Закрыть вручную                   |
| `SET_BREAKEVEN`| 32       | Перенести стоп в безубыток        |
| `HIT_TP`       | 64       | Сообщение о достижении TP         |
| `HIT_SL`       | 128      | Сообщение о срабатывании SL       |
| `LIMIT`        | 256      | Лимитный ордер                    |
| `ACTIVATED`    | 512      | Лимит активирован                 |
| `CANCELLED`    | 1024     | Ордер отменён                     |
| `EXTRA`        | 2048     | Дополнительный вход               |

## Обработка в реальном времени (`CheckTick`)

Метод вызывается на каждом тике и обрабатывает сообщения немедленно, не дожидаясь закрытия бара.

- Использует упорядоченный список `m_MessagesList` (построен один раз при инициализации) и курсор `m_NextMessageIndex` — O(1) быстрый путь.
- Если очередное сообщение в списке имеет время ≤ времени тика, оно передаётся в `HandleSignalAction`, а `m_NextMessageIndex` продвигается вперёд.
- Обработанные здесь сообщения **не повторяются** в `ProcessSetup`.

---

## Логика разбора на каждом баре (`ProcessSetup`)

Метод вызывается на каждом новом закрытом баре и выполняет два шага.

### Шаг 1 — Проверка цены (price-movement check)

Для каждой сделки из `m_MessageSignalArgsMap`:

- **Ожидающий лимитный ордер** (`IsLimit && !IsActive`):
  - BUY-лимит активируется, когда `low бара <= entry`; SELL-лимит — когда `high бара >= entry`.
  - Если TP достигнут раньше входа — ордер **отменяется** (`OnCanceled`), сделка удаляется.
  - При активации помечается `IsActive = true` и переходит к обычной проверке TP/SL.
- **Активная сделка**:
  - Если **high бара > TP** (лонг) или **low бара < TP** (шорт) → событие `TakeProfit`, сделка удаляется.
  - Если **low бара < SL** (лонг) или **high бара > SL** (шорт) → событие `StopLoss`, сделка удаляется.

> Эта логика работает независимо от наличия ответных сообщений в канале.

### Шаг 2 — Обработка новых сообщений

С помощью бинарного поиска (`LowerBoundAfter`, O(log N)) находится первый индекс сообщения с временем > `prevBarDateTime`. Берётся максимум этого индекса и `m_NextMessageIndex` (чтобы не повторить то, что уже обработал `CheckTick`), и все сообщения с временем ≤ `barDateTime` передаются в `HandleSignalAction`.

---

## Логика `HandleSignalAction` (приоритет обработки)

```
ProcessMessage(сообщение) → битовая маска res
```

Приоритет обработки **в порядке убывания** (первое совпавшее условие выполняется, остальные — нет):

1. **ENTER_BUY / ENTER_SELL + SET_TP + SET_SL** → открыть новую сделку, добавить в `m_MessageSignalArgsMap`.  
   - Ключевые лимитные ордера (`LIMIT` флаг из текста) — **пропускаются** (`return`).
   - Выбирается TP **ближайший к цене входа** (не первый в списке).
   - Если текущая цена невыгоднее цены входа и `useLimitOrders = true` → сделка открывается как лимитная (`isLimit: true`).
   - ID сигнала сохраняется в `m_LastOpenedSignalId`.

2. **EXTRA** → повторный вход с теми же TP/SL что у исходного сигнала.

3. **SET_BREAKEVEN** → перенос стопа в безубыток (на уровень входа).
   - Если исходный сигнал — ожидающий лимит (`IsLimit && !IsActive`): лимитный ордер **отменяется** (`OnCanceled`), сигнал удаляется из карты.
   - Иначе (только если ещё не было безубытка): вызывается `OnBreakEven` с флагами `hasBreakeven: true, closeHalf: true`, SL обновляется. Сигнал **остаётся** в карте.

4. **HIT_TP** → событие TakeProfit по сообщению канала, сигнал удаляется.

5. **HIT_SL** → событие StopLoss по сообщению канала, сигнал удаляется.

6. **SET_SL / SET_TP** → скорректировать стоп/тейк:
   - SET_SL обновляет SL у всех сделок с тем же `Comment`, **пропуская** те, у которых уже выставлен безубыток (`HasBreakeven`).
   - SET_TP: выбирается TP ближайший к цене входа исходного сигнала.

7. **CLOSE** → ручное закрытие, сигнал удаляется из карты. Игнорируется, если в сообщении также есть `\bhold\b`.

8. **ACTIVATED** → активация лимитного ордера (`IsActive = true`).

9. **CANCELLED** → отмена ожидающего лимита, сигнал удаляется.

### Связь «ответ — исходный сигнал»

Если у сообщения есть `reply_to_msg_id` — ищем сигнал в карте по этому ID.  
Если нет — используем `m_LastOpenedSignalId` (ID последнего открытого входа; ранее брался `LastOrDefault()` из всей карты).

## Разбор входного сигнала (алгоритм `ProcessMessage`)

### Цена входа
- Берётся из `SIGNAL_REGEX` (группа 3) через разделитель `@` или `-`.  
- Если не распознана — используется `ask` (лонг) или `bid` (шорт) от `ITradeViewManager`.

### Стоп-лосс
- Берётся из `SL_REGEX` (поддерживаются целые числа и числа с десятичной точкой).
- Дополнительно считается **дефолтный стоп** по словарю `DEFAULT_SL_PIPS` (для XAUUSD = 1000 пунктов от цены входа).
- Распознанный SL принимается только если **отклоняется от дефолтного не более чем на 20%** (`m_MaxStopRatio = 0.20`) и помечается флагом `HasExplicitSl = true`.
- Если SL в тексте не найден — используется дефолтный (если отклонение от текущей цены bid ≤ 20%).
- **Сигнал пропускается, если нет SL** (ни явного, ни дефолтного): вход требует наличия обоих — `hasTp && hasSl`.

### Тейкпрофит
- Берётся из всех совпадений `TP_REGEX` (список TP1, TP2, …).  
- При открытии сделки используется **TP, ближайший к цене входа** (не первый в списке).
- Если ни одного TP не найдено — дефолт из `DEFAULT_TP_PIPS` (для XAUUSD = 1000 пунктов).

### Валидация сигнала
Если одновременно выставлены `ENTER_BUY` и `ENTER_SELL`, или TP/SL не соответствуют направлению — сигнал отбрасывается с `SignalAction.DEFAULT`.

## Привязка к барам (`BarPoint`)

Все `BarPoint` (уровни входа, TP, SL, BE) создаются с временем **`LastBarOpenDateTime`** — временем открытия последнего бара на момент обработки. Использовать `signal.DateTime` (время сообщения в Telegram) **нельзя**: в cTrader `BarPoint` требует время, которое существует в индексе баров, иначе `GetIndexByTime` выбросит исключение.

---

## Нормализация DateTime

| Источник                   | Исходный Kind              | Что делается                                                    |
|----------------------------|----------------------------|-----------------------------------------------------------------|
| JSON-поле `date`           | `Local` (Newtonsoft default) | `DateTime.SpecifyKind(date.ToUniversalTime(), Unspecified)`  |
| Время открытия бара (cTrader) | `Unspecified` (UTC-тики) | Используется как есть, `.ToUniversalTime()` **не вызывается** |
| Время открытия бара (тесты) | `Utc`                     | `DateTime.SpecifyKind(time, DateTimeKind.Utc)` при загрузке CSV |

> **Почему нельзя вызывать `.ToUniversalTime()` на барах cTrader:**  
> cTrader возвращает времена с `DateTimeKind.Unspecified`, но значение тиков уже UTC. `.ToUniversalTime()` на `Unspecified` предполагает машинный часовой пояс и даёт неверный результат на серверах не в UTC.
