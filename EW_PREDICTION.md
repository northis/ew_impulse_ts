# Алгоритм предсказания волн Эллиотта — проектный документ

Расширяет движок разметки (`ElliottWaveExactMarkup`) **режимом предсказания**:
диапазон анализа доходит до последнего закрытого бара, незавершённые модели
порождают проекции ценовых целей, отображаемые пунктирными линиями.

Связанные документы:
- `EW_MARKUP.md` — алгоритм разметки завершённых диапазонов (v1).
- `EW_RULES.md` — каталог моделей, карты Фибоначчи, правила генерации.

## 1. Цели и ограничения

| # | Требование |
|---|---|
| G1 | Анализировать от наиболее удалённого экстремума (в последних N барах) до **последнего закрытого бара**, а не до ближайшего контрэкстремума. |
| G2 | Разрешить модели, в которых **последние 1–2 волны отсутствуют** (незавершённые). Проецировать их конечные точки с помощью соотношений Фибоначчи и образующих линий. |
| G3 | Обновлять строго **по закрытию бара** — внутрибарные изменения цены игнорируются. |
| G4 | Рисовать подтверждённые волны сплошными линиями, проецируемые — **пунктирными**. |
| G5 | Существующий API `Parse(points)` остаётся неизменным; `AutoMarkupZigzagBaseIndicator` и аналогичные потребители работают с завершёнными сегментами, как и прежде. |
| G6 | На текущем этапе считаем **только наиболее вероятный вариант** разметки. Архитектура должна позволять в будущем возвращать несколько вариантов, но сейчас — один лучший. |

## 2. Выбор диапазона

### 2.1 Текущее поведение (`IterativeElliottWaveExactIndicator`)

1. Берём последние `N` баров.
2. Находим глобальный максимум и минимум по индексам баров.
3. Наиболее удалённый (старший) экстремум → `startPoint`; ближайший (младший) → `endPoint`.
4. Строим зигзаг между `startPoint` и `endPoint`.
5. Передаём зигзаг в `Parse(points)` — участвуют только **завершённые** сегменты.

### 2.2 Диапазон предсказания

1. Берём последние `N` баров.
2. Находим глобальный максимум и минимум. **Наиболее удалённый** (старший) экстремум —
   `startPoint`.
3. `endPoint` = **последний закрытый бар** (бар с индексом `Bars.Count - 2`,
   когда текущий бар ещё формируется, или `Bars.Count - 1` по событию закрытия бара).
4. Направление: `isUp = endPoint.Value > startPoint.Value` — по цене закрытия
   `endPoint` относительно цены экстремума `startPoint`.
5. Строим внутренний зигзаг от `startPoint` до `endPoint` обычным способом.
6. Последний сегмент зигзага может закончиться **раньше** `endPoint` — цена
   между последним разворотом и закрытием является **активным сегментом**,
   ещё не сформировавшим подтверждённый экстремум.

### 2.3 Построение активного сегмента

После построения зигзага добавляем точку на `endPoint`, если последний разворот
не совпадает с последним баром:

```
lastPivot = innerPoints[^1]
if lastPivot.BarIndex < endBarIndex:
    activePrice = isUp direction from lastPivot
        ? BarProvider.GetHigh(endBarIndex)   // восходящий активный сегмент
        : BarProvider.GetLow(endBarIndex)    // нисходящий активный сегмент
    innerPoints.Add(BarPoint(activePrice, endBarIndex))
    пометить эту точку как "активную" (неподтверждённую)
```

Проходы `RefineToCorridors` / `EndFixCorridors` выполняются по-прежнему,
но активная точка освобождена от end-fix (это живая цена, а не подтверждённый
разворот).

## 3. Сопоставление неполных моделей

### 3.1 Концепция

Существующий `Parse` работает с `S` сегментами и требует `S ≥ K` (где `K` =
ожидаемое число волн). Для предсказания мы также принимаем `S = K − 1` и
`S = K − 2` (не хватает одной или двух последних волн).

**Уровень завершённости** `C = S / K`:
- `C = 1.0` → все волны присутствуют (идентично существующему алгоритму).
- `C = (K−1)/K` → не хватает последней волны (напр., импульс с 4 из 5 сегментов).
- `C = (K−2)/K` → не хватает двух последних волн (напр., импульс с 3 из 5 сегментов).

Модели с `C < (K−2)/K` не рассматриваются — слишком мало подтверждённых волн
для надёжной идентификации.

### 3.2 Генерация частичных кандидатов

Расширяем цикл перебора комбинаций в `ParseInternal`:

```
for each model in targetModels:
    K = ExpectedWaves(model)
    if S >= K:
        // существующая логика: полное совпадение + хвост
        ...
    if S == K - 1:
        // кандидат: волны 1..K-1 подтверждены, волна K проецируется
        waves = BuildWaves(segments, count: S)
        if HardRulesCheck(model, waves, partial: true) == null:
            projection = ProjectNextWave(model, waves)
            yield CandidateWithProjection(model, waves, [projection])
    if S == K - 2 and K >= 5:
        // кандидат: волны 1..K-2 подтверждены, волны K-1 и K проецируются
        waves = BuildWaves(segments, count: S)
        if HardRulesCheck(model, waves, partial: true) == null:
            projections = ProjectNextTwoWaves(model, waves)
            yield CandidateWithProjection(model, waves, projections)
```

**Важно:** на текущем этапе из всех частичных кандидатов выбирается **один
лучший** по скорингу (§5). Архитектура позволяет вернуть список, но пока
возвращается единственный результат.

### 3.3 Предположение растяжения (extension)

Если набор подтверждённых волн не укладывается ни в одну модель целиком,
алгоритм пробует интерпретировать последние волны как **растяжение** одной
из волн старшей модели. Например:

- Подтверждены: импульс — зигзаг — зигзаг. Ни одна модель не даёт полного
  совпадения.
- Предположение: второй зигзаг — это волны 1–2 растяжения волны c (из тройки
  a-b-c), и мы ожидаем ещё волны 3–5 of c.
- Итог: проекция строится на старшем уровне (a-b-c), а растяжение даёт
  +1 вложенный уровень для спроецированной волны c.

Такой подход позволяет перебирать комбинации волн с учётом проекций: когда
невозможно разметить участок как завершённую модель, считаем, что просто не
хватает волн, и строим прогноз по максимально вероятному скорингу. Сложность
при этом не растёт значительно, так как рекурсивное предсказание ограничено
одним дополнительным уровнем вложенности.

### 3.4 Ослабление жёстких правил для частичных кандидатов

| Правило | Корректировка для частичных кандидатов |
|---|---|
| §4.1 ценовые правила | Применяются только к **подтверждённым** волнам. Отсутствующие волны не проверяются по ценовым ограничениям. |
| §4.1-непрерывность | Подтверждённая часть должна покрывать от `altPoints[0]` до последней доступной точки без разрывов. Проецируемый хвост освобождён. |
| §4.1-свеча | Проверяется только для подтверждённых волн. |
| §4.1-конечная точка | Пропускается для активного сегмента (его конец — не подтверждённый экстремум). |
| W3 ≥ max(W1,W5) | Если W5 проецируется — проекция строится с учётом этого правила. Если W3 — последняя подтверждённая, проверка откладывается. |
| Правила хвоста (§5) | Не применяются — за спроецированной волной нет хвоста. |

### 3.5 Активный сегмент как последняя подтверждённая волна

Когда активный сегмент (§2.3) формирует K-ю волну модели, кандидат
считается **завершённым, но неподтверждённым**: все K волн присутствуют в
списке точек, но конечная точка последней волны помечена как `active`. Это
более сильный кандидат, чем с по-настоящему отсутствующей волной, так как
ценовые правила можно частично проверить. Проекция в этом случае — **целевой
диапазон** завершения волны, а не полностью новая волна.

### 3.6 Боковое движение (нет подходящей модели)

Если активный сегмент многократно меняет направление и ни одна модель
(включая частичные) не набирает достаточного скоринга, разметка не строится.
Это нормальная ситуация для бокового движения. В таких случаях предполагаем,
что рынок находится в **боковой коррекции** — треугольник, флет или комбинация.
Как только сформируется достаточно разворотных точек, одна из этих моделей
начнёт соответствовать, и проекция появится.

## 4. Цели проекции

### 4.1 Одна недостающая волна

При наличии подтверждённых волн `W1..W(K-1)` проецируем `W(K)`:

1. Определяем применимую карту Фибоначчи для отношения недостающей волны к
   эталонной (из `EW_RULES.md` §4):
   - Импульс W5 → `IMPULSE_5_TO_1` (W5/W1).
   - Зигзаг C → `ZIGZAG_C_TO_A` (C/A).
   - Диагональ W5 → `CONTRACTING_DIAGONAL_3_TO_1` (с корректировкой границ).
   - Флет C → `MAP_EX_FLAT_WAVE_C_TO_A` / `MAP_REG_FLAT_WAVE_C_TO_A`.
   - Треугольник E → отношение к D.

2. Вычисляем проецируемую конечную точку по **наиболее вероятному** уровню
   Фибоначчи (максимальный вес в карте):
   ```
   projectedLength = referenceWaveLength × fibRatio
   projectedEnd    = lastConfirmedEnd ± projectedLength  (± по направлению)
   ```

3. Проекция содержит **оценку числа баров** по правилам соразмерности
   длительностей (§20 `EW_RULES.md`):
   ```
   estimatedBars = referenceBars × durationCoeff
   ```
   Это приблизительная оценка, используемая для длины линии проекции.
   Подробнее о соразмерности — см. §4.3.

### 4.2 Две недостающие волны

Когда `S = K − 2` (напр., импульс после волны 3):

1. Проецируем `W(K-1)` — обычно коррекционная волна (волна 4 в импульсе).
   Используем карту глубины коррекции (`MAP_SHALLOW_CORRECTION` или
   `MAP_DEEP_CORRECTION` в зависимости от правил модели и чередования
   с волной 2).
2. Из спроецированной конечной точки `W(K-1)` проецируем `W(K)` по §4.1.
3. Выбираем **один наиболее вероятный путь** — комбинацию уровней с
   максимальным произведением весов Фибоначчи.

В будущем можно вернуть несколько путей (дерево проекций), но на текущем
этапе — только один лучший.

### 4.3 Соразмерность длительностей

Помимо ценовых соотношений Фибоначчи, для оценки длительности проецируемых
волн используется **соразмерность** подтверждённых волн:

- **Импульс, волны 1–3 подтверждены:** волна 4 предполагается примерно такой же
  по длительности или чуть длиннее, чем волна 2 (согласно правилам EW). Зная
  длительность W2, приблизительно рассчитываем, где во времени закончится W4,
  а затем и W5.
- **Зигзаг, волны a–b подтверждены:** по их соотношению и длительности
  оцениваем область завершения волны c.
- **Общее правило:** `estimatedBars(Wn) ≈ referenceBars(Wref) × durationCoeff`,
  где `durationCoeff` берётся из распределения длительностей соответствующей
  модели.

### 4.4 Образующие линии (trendlines)

Для ряда моделей проекция опирается не (только) на соотношения Фибоначчи,
а на **образующие линии** — трендовые, проведённые через ключевые точки:

| Модель | Образующая 1 | Образующая 2 | Применение |
|---|---|---|---|
| Сужающийся треугольник | Линия начало–B | Линия A–C | Цели по волнам D и E — пересечение с образующими |
| Бегущий треугольник | — | Линия A–C | Только A–C, так как волна B обычно выбивается за первую образующую |
| Расходящийся треугольник | Линия начало–B | Линия A–C | Цели D и E — пересечение с расходящимися образующими |
| Диагональ (начальная/конечная) | Линия 1–3 | Линия 2–4 | Цели по волнам 3(если предсказываем)–5 |
| Плоскость (flat) | — | — | Волна B ≈ 100–127% от A; волна C — по Фибо от A (см. §4.1) |

Для **плоскости**: волна B должна чуть зайти за начало A, обычно останавливаясь
на ~127% от размера волны A. Зная это, из подтверждённых волн a–b рассчитываем
ожидаемое окончание волны C.

Для **треугольников**: как только подтверждены волны A–C, строим образующие
линии и проецируем точки пересечения для волн D и E. Это геометрический расчёт,
дополняющий оценку по Фибоначчи.

Для **диагоналей**: линии 1–3 и 2–4 работают аналогично образующим треугольника.

### 4.5 Структура данных проекции

```csharp
record WaveProjection(
    int WaveIndex,             // позиция 1-based (напр., 5 для W5)
    string WaveKey,            // "5", "c", "e" и т.д.
    double ProjectedValue,     // ценовая цель
    int EstimatedBarIndex,     // приблизительный индекс бара
    double FibRatio,           // использованный уровень Фибоначчи
    double FibWeight,          // вес из карты (прокси уверенности)
    bool UsedTrendline         // true, если цель определена по образующей
);

record PredictionResult(
    ExactParsedNode ConfirmedNode,       // подтверждённые волны (существующее дерево)
    WaveProjection[] Projections,        // 1-2 проецируемые волны
    double ConfirmedScore,               // скоринг подтверждённой части
    double CompletionLevel               // K-подтверждённых / K-всего
);
```

## 5. Корректировка скоринга

### 5.1 Частичный скоринг Фибоначчи

Только подтверждённые пары волн участвуют в `fibScore`. Для недостающих волн
используются **только доступные соотношения**:

```
fibScore = ∏ fibScore_i  для i ∈ подтверждённых пар волн
```

Например, если подтверждены волны a и b зигзага — вычисляется `fibScore(b/a)`;
соотношение c/a не участвует, так как волна c спроецирована.

### 5.2 Штраф за неполноту

Неполные модели получают мультипликативный штраф, чтобы кандидат 4-из-5
никогда не обогнал идеальное совпадение 5-из-5 при том же качестве Фибоначчи:

```
completionPenalty = completionLevel ^ 2
```

| Завершённость | Штраф |
|---|---|
| 5/5 = 1.00 | 1.000 |
| 4/5 = 0.80 | 0.640 |
| 3/5 = 0.60 | 0.360 |
| 2/3 = 0.67 | 0.444 |

### 5.3 Бонус активного сегмента

Если активный сегмент (§3.5) формирует последнюю волну и текущая цена уже
достигла минимального уровня Фибоначчи для этой волны, присваиваем частичный
Фибо-скоринг:

```
activeFibScore = weight(closestFibAboveMinimum) / sumWeights
                 × min(1.0, currentRatio / targetRatio)
```

Вознаграждает кандидатов, где рынок уже значительно продвинулся в зону
проецируемой волны.

### 5.4 Приоритет подтверждённых моделей

Когда существуют и полные, и неполные кандидаты, **полные модели имеют приоритет**
при прочих равных. `completionPenalty` естественным образом ставит неполные
модели ниже равных по качеству полных. Если ни одна полная модель не прошла
жёсткие правила, а неполная прошла — возвращается неполная.

Дополнительно: если подтверждённые волны можно интерпретировать как растяжение
(§3.3), кандидат с растяжением получает скоринг на уровне модели старшего порядка,
что может дать ему преимущество перед простым частичным кандидатом.

### 5.5 Комбинированный скоринг

```
rawScore = baseScore × fibScore × durationScore × completionPenalty
           × activeBonus                            // §5.3, если применимо
adjustedScore = rawScore × (1 + depthCoverage × DEPTH_COVERAGE_BONUS)
```

## 6. Цикл обновления по закрытию бара

### 6.1 Триггер

Индикатор пересчитывается **только когда `IsLastBar == true` и индекс бара
продвинулся** (новый бар закрылся). Отслеживается сравнением `index` с
`m_LastCalculatedIndex`:

```csharp
if (!IsLastBar) return;
if (index == m_LastCalculatedIndex) return;
m_LastCalculatedIndex = index;
```

### 6.2 Шаги пересчёта

По закрытию каждого бара:

1. Пересчитать `startPoint` (наиболее удалённый экстремум за последние N баров) —
   глобальный экстремум может сместиться при продвижении окна.
2. Установить `endPoint` на последний закрытый бар.
3. Перестроить внутренний зигзаг.
4. Запустить `ParsePredictive(points)` — расширенный парсинг, принимающий
   частичные модели.
5. Сравнить новый лучший результат с кэшированным:
   - Если тип модели и структура подтверждённых волн **идентичны** (те же
     конечные точки сегментов), обновить только проекцию и данные активного
     сегмента.
   - Иначе — очистить все объекты графика и перерисовать.
6. Кэшировать новый результат.

### 6.3 Инвалидация

Разметка инвалидируется (полная перерисовка) когда:
- Глобальный экстремум сместился на другой бар.
- Ранее спроецированная волна теперь подтверждена (появился новый разворот зигзага).
- Активный сегмент сменил направление, разрушив кандидата.
- Цена нарушила жёсткое правило текущей лучшей модели (напр., волна 4
  проникла в территорию волны 1 в импульсе).

## 7. Отрисовка

### 7.1 Подтверждённые волны

Отрисовываются как в существующем `DrawMarkupNode`:
- Сплошные линии тренда с цветом по типу родительской модели.
- Стопкой наложенные метки с обозначением степени.

### 7.2 Проецируемые волны

| Свойство | Значение |
|---|---|
| Стиль линии | `LineStyle.DotsRare` (пунктир) |
| Цвет линии | Тот же, что у подтверждённых волн, 50% прозрачности |
| Суффикс метки | `"?"` добавляется (напр., `(v)?`, `c?`) |
| Стиль метки | Тот же цвет, 50% прозрачности |

Проецируемые волны начинаются от конечной точки последней подтверждённой волны
и заканчиваются на спроецированной цене / расчётном индексе бара.

### 7.3 Целевые зоны

Для проекции опционально рисуется **горизонтальный прямоугольник** на целевой
цене ± один уровень Фибоначчи для обозначения вероятной зоны завершения.
По умолчанию отключено; активируется параметром индикатора `ShowTargetZones`.

### 7.4 Образующие линии

Если цель определена по образующей (§4.4), линия образующей рисуется тонким
пунктиром того же цвета с 30% прозрачности.

### 7.5 Альтернативные варианты (будущее)

На текущем этапе рисуется **только лучший** вариант предсказания. В будущем
параметр `ShowAlternatives` (макс. 3) позволит рисовать дополнительные варианты
с убывающей прозрачностью (70%, 50%, 30%). Архитектура `PredictionResult` уже
поддерживает список кандидатов.

## 8. API и обратная совместимость

### 8.1 Существующий API — без изменений

```csharp
// Разметка завершённого диапазона (v1) — без изменений
List<ExactParsedNode> Parse(List<BarPoint> points)
```

Все текущие потребители (`AutoMarkupZigzagBaseIndicator`,
`IterativeElliottWaveExactIndicator` в режиме завершённого диапазона, юнит-тесты)
вызывают `Parse()` и получают только полностью совпавшие модели.
**Этот метод не меняется.**

### 8.2 Новый API

```csharp
/// <summary>
/// Парсинг в режиме предсказания. Возвращает лучший результат, который
/// может быть как завершённой, так и незавершённой моделью с проекциями.
/// </summary>
/// <param name="points">Экстремумы зигзага, включая активную конечную точку.</param>
/// <param name="activeBarIndex">
/// Индекс бара живой (неподтверждённой) конечной точки. Точки на этом индексе
/// считаются активными — жёсткие правила ослаблены для сегмента, касающегося
/// этой точки.
/// Передать -1 для отключения режима предсказания (эквивалентно <see cref="Parse"/>).
/// </param>
PredictionResult ParsePredictive(List<BarPoint> points, int activeBarIndex)
```

Внутри `ParsePredictive` вызывает существующий `ParseInternal` для полных
кандидатов и новый `ParsePartialInternal` для неполных. Результаты объединяются
в единый ранжированный список, из которого возвращается лучший.

> **Расширяемость:** сигнатура может быть изменена на
> `List<PredictionResult>` в будущем, когда потребуется возвращать несколько
> вариантов.

### 8.3 Новый индикатор

`PredictiveElliottWaveIndicator` — новый класс индикатора, наследующий от
`ElliottWaveIndicatorBase`. Параметры:

| Параметр | Умолчание | Описание |
|---|---|---|
| `BarsCount` | 200 | Окно просмотра |
| `MarkupDepth` | `MAX_MARKUP_DEPTH` | Глубина рекурсии |
| `ShowTargetZones` | `false` | Рисовать прямоугольники целевых зон Фибоначчи |

Существующий `IterativeElliottWaveExactIndicator` **не модифицируется**.
Он сохраняет поведение завершённого диапазона. Пользователь выбирает,
какой индикатор загрузить.

## 9. Известные ограничения и решения

### 9.1 Сложность: один вариант вместо дерева

**Ограничение:** полный перебор всех комбинаций неполных моделей с деревом
проекций порождает взрыв вариантов, особенно при `S = K − 2` с рекурсией.

**Решение:** на текущем этапе считаем **только один наиболее вероятный вариант**.
Архитектура (`PredictionResult`, ранжирование кандидатов) уже закладывает
поддержку нескольких вариантов, но отдаём единственный лучший. В будущем
можно включить top-N без существенной переработки.

### 9.2 Боковое движение (нет варианта разметки)

**Ограничение:** при частых сменах направления активного сегмента ни одна модель
может не набрать достаточного скоринга.

**Решение:** это нормальная ситуация. В боковом движении предполагаем
**боковую коррекцию** — треугольник, флет или комбинацию. Как только
сформируется достаточно разворотных точек, модель начнёт соответствовать.
Отсутствие разметки — допустимый результат.

### 9.3 Фибо-карты и доступные волны

**Ограничение:** карты Фибоначчи описывают соотношения **завершённых** волн.

**Решение:** используем только ту часть соотношений, которая задействует
**доступные (подтверждённые) волны**. Например, если подтверждены волны a и b,
по ним рассчитываем область завершения волны c через `ZIGZAG_C_TO_A`. Не
используем соотношения, требующие ещё не существующих волн.

### 9.4 Оценка длительности через соразмерность

**Ограничение:** оценка по барам неточна, так как правила длительности имеют
широкие допуски.

**Решение:** используем **соразмерность волн**. Если подтверждены волны 1–3
импульса, предполагаем, что волна 4 будет примерно такой же или чуть длиннее
по длительности, чем волна 2. Соответственно, примерно рассчитываем, где
рисовать проекции W4 и W5. Оценку длительности помечаем в UI как
приблизительную (ориентир для длины линии, а не точный прогноз времени).

### 9.5 Проекции треугольников и плоскостей через образующие

**Ограничение:** у треугольников и плоскостей немонотонная волновая структура,
и простые соотношения Фибоначчи недостаточны.

**Решение:**
- **Треугольник (сужающийся):** цели по волнам D и E определяются пересечением
  с **образующими линиями** — линия от начала треугольника до B и линия A–C.
- **Бегущий треугольник:** только линия A–C (волна B обычно выбивается).
- **Расходящийся треугольник:** образующие расходятся — аналогичный принцип.
- **Плоскость:** волна B ≈ 100–127% от A, волна C — по Фибо от A.
- **Диагонали:** линии 1–3 и 2–4 работают как образующие треугольника.

Если пересечение с образующей не даёт валидной проекции (напр., нет пересечения
в будущем), частичный кандидат отбрасывается.

### 9.6 Приоритет подтверждённых моделей и растяжения

**Ограничение:** набор подтверждённых волн может не укладываться ни в одну
модель.

**Решение:** перебор комбинаций с учётом **растяжения** (§3.3). Если прямое
соответствие не найдено, пробуем интерпретировать последние волны как начало
растяжения. Это даёт +1 уровень вложенности, но не более, что удерживает
сложность в допустимых пределах. Подтверждённые модели (без допущений о
растяжении) имеют приоритет при равном скоринге.

### 9.7 Предсказание только старшего уровня

**Ограничение:** рекурсивное предсказание подволн спроецированной волны
порождает спекулятивные Фибо-соотношения на двух уровнях одновременно.

**Решение:** предсказываем **только старший уровень разметки**. Исключение —
растяжение (§3.3), где допускается +1 вложенный уровень для спроецированной
волны. Полностью рекурсивное предсказание подволн проекции откладывается
на v2.
