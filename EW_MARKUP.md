## 1. Выбор участка для разметки

Индикатор принимает параметр `N` — глубина в свечах. Алгоритм:

1. Берём последние `N` баров.
2. Находим глобальный **максимум** и **минимум** этого диапазона (два «граничных» экстремума).
3. Определяем, какой из двух граничных экстремумов **ближе к последнему бару** — назовём его `nearExtremum`.
4. Между `nearExtremum` и последним баром ищем **контр-экстремум** (первый экстремум, направленный в противоположную сторону от `nearExtremum`) — назовём его `counterExtremum`.
5. **Участок разметки**: от `nearExtremum` до `counterExtremum`.

> Пример (бычий рынок, последний бар у максимума):
> - Глобальный максимум — ближний к финалу (`nearExtremum`).
> - Между ним и концом — первый локальный минимум (`counterExtremum`).
> - Размечаем участок максимум → минимум как нисходящую волну.

**Что закладывается на будущее (v2):** кроме описанного участка, возможен режим
«разметка до последнего бара» — тогда `counterExtremum` заменяется последним баром.
Логика идентична, просто последний бар используется как незавершённый экстремум.

## 2. Препроцессинг: построение «цепочки сегментов»

Цель — получить набор **элементарных колебаний** (up/down отрезков), из которых
алгоритм будет собирать модели.

1. К участку разметки применяем `SimpleExtremumFinder` с периодом **0.01** (минимальная
   амплитуда отклонения = 1 % от диапазона).
2. Результат — список `BarPoint[]` чередующихся минимумов/максимумов.
3. Из этих точек строим **список сегментов** (далее — просто `segments`):
   каждый сегмент = пара (start: `BarPoint`, end: `BarPoint`).

Число сегментов `S = len(segments)`.
Каждая подволна любой модели соответствует одному **или нескольким** элементарным сегментам.

> **Ключевое соглашение:** в v1 каждая подволна = ровно один сегмент.
> Таким образом, импульс требует не менее 5 сегментов, зигзаг — 3, треугольник — 5 и т.д.

## 3. Главный метод разметки

```
Parse(
    segments         : List<Segment>,
    targetModel      : ElliottModelType?,     // null → перебор всех возможных
    parentContext    : WaveContext?            // null → нет ограничений сверху
) → List<ParsedModel>                         // отсортировано по убыванию Score
```

### Логика перебора

1. Формируем список кандидатов для проверки:
   - если `targetModel != null` → только эта модель;
   - иначе → `TargetModels` (см. `ElliottWaveExactMarkup.TargetModels`).

2. Для каждой модели из кандидатов:
   a. Определяем `K = ExpectedWaves(model)` — требуемое число подволн.
   b. Если `S < K` → **пропуск** (не хватает сегментов).
   c. Иначе пробуем два варианта разбивки:
      - **«Полная посадка»**: берём ровно `K` сегментов (сегменты 0..K-1),
        модель занимает весь доступный диапазон. Приоритет выше.
      - **«С хвостом»**: пробуем все K-размерные окна, начиная с позиции 0
        (т.е. модель занимает сегменты 0..K-1, сегменты K..S-1 образуют хвост).
        В v1 рассматривается только один такой вариант (минимальный хвост).

3. Для каждого кандидата считаем **Score** (см. §6).

4. Проверяем `parentContext` — дополнительные ограничения (§7).

5. Добавляем в результирующий список `ParsedModel` с рассчитанным Score.

6. Сортируем по убыванию Score и возвращаем.

## 4. Поиск модели в цепочке сегментов

### 4.1 Проверка базовых ценовых правил

Для каждого кандидата с `K` сегментами проверяем hard-правила (нарушение → Score = 0,
модель исключается):

| Модель | Ключевые hard-правила |
|---|---|
| IMPULSE | W3 ≥ max(W1, W5); W4 не пересекает территорию W1; |
| DIAGONAL (сходящийся) | каждая волна короче предыдущей одноимённой; W4 пересекает W1 |
| DIAGONAL (расходящийся) | каждая волна длиннее предыдущей |
| ZIGZAG | C ≥ 0.618 × A; B ≤ 0.95 × A |
| FLAT_EXTENDED | B ≥ A (выходит за начало); C ≥ 1.618 × A |
| FLAT_RUNNING | B ≥ A; C ≤ 1.618 × A |
| TRIANGLE_CONTRACTING | каждая волна короче предыдущей |
| TRIANGLE_RUNNING | то же + B превышает начало треугольника |

### 4.2 Допустимые sub-модели

Каждая подволна в v1 — `SIMPLE_IMPULSE` (один сегмент).
Проверять, подходит ли `SIMPLE_IMPULSE` как sub-модель для данной позиции, можно через
`ElliottWavePatternHelper.ModelRules` — `SIMPLE_IMPULSE` разрешён везде.

### 4.3 Идентификация «расширенной» волны / ведущей волны

Для импульса — дополнительно выясняем:
- W3 / W1 > 1.0 → W3 расширенная (стандартный случай).
- W1 > W3 (W1 расширенная) → допустимо, но более редко.
- W5 > W3 → W5 расширенная.

Это влияет на скоринг по картам Фибоначчи, но не является hard-ограничением.

## 5. Правила «хвоста»

Если модель занимает только первые `K` из `S` сегментов, остаток (`tail`) может
аннулировать найденную модель или снизить её Score.

### Правила отмены по хвосту

| Модель | Свойство хвоста, отменяющее модель |
|---|---|
| TRIANGLE_CONTRACTING | Первый сегмент хвоста должен двигаться в направлении «thrust» (выброса из треугольника). Если он идёт против ожидаемого thrust-а, модель ОТМЕНЯЕТСЯ. |
| TRIANGLE_RUNNING | То же, что для TRIANGLE_CONTRACTING. |
| IMPULSE | Первый сегмент хвоста не должен уходить глубже старта W4 (иначе структура нарушена). |
| DIAGONAL_CONTRACTING_ENDING | Первый сегмент хвоста должен развернуться мощно и быстро (sharp reversal). Медленный хвост снижает Score. |
| ZIGZAG / FLAT | Хвост не отменяет модель, только снижает Score (нет завершённого следующего движения). |

### Реализация правил хвоста в C# — предложение

Задать правила как лямбды/делегаты в словаре вида:

```csharp
static readonly Dictionary<ElliottModelType, Func<Segment[], int, TailRuleResult>> TailRules =
new()
{
    [ElliottModelType.TRIANGLE_CONTRACTING] = (segs, modelEnd) =>
    {
        var thrust = segs[modelEnd];
        bool correctDirection = IsThrust(segs, modelEnd, expectedUp: segs[modelEnd - 1].IsUp);
        return correctDirection ? TailRuleResult.Valid : TailRuleResult.Rejected;
    },
    // ...
};
```

Это позволяет легко добавлять/менять правила для каждой модели изолированно.

## 6. Скоринг

Score — вещественное число в диапазоне (0, 1], произведение множества компонентов.
Итоговый Score = ∏ всех компонентов.

### 6.1 Базовая вероятность модели

Из таблицы §2 EW_RULES.md: `modelCoeff / 1.0` (нормализуется к 1.0 как максимуму).

```
baseScore = modelProbabilityCoeff / MaxModelCoeff   // MaxModelCoeff = 1.0
```

### 6.2 Соответствие уровням Фибоначчи

Для каждой пары (волна i+1 / волна i) вычисляем реальное отношение длин и сравниваем с
ближайшим уровнем в соответствующей карте Фибоначчи (§4 EW_RULES.md).

```
fibScore_i = weight(nearestFibLevel) / sum(allWeights)
           × exp(−k × |actualRatio − nearestFibLevel| / nearestFibLevel)
```

где `k ≈ 5` управляет «жёсткостью» штрафа за отклонение от уровня.

Общий `fibScore = ∏ fibScore_i`.

### 6.3 Соразмерность продолжительности волн (§20 EW_RULES.md)

Для импульса (§20.1):

```
durationScore = durationPenalty(W2, W4, limit_4over2=1.5, limit_2over4=1.1)
```

Для треугольников и диагоналей — геометрический score (§20.2–20.5):
```
geoScore = exp(−k × distanceToLine / patternRange)
```

### 6.4 Бонус «полной посадки»

Если модель заняла все `S` сегментов без хвоста:
```
fullFitBonus = 1.2
```
(коэффициент > 1 повышает Score целиком; ограничить результат 1.0 при нормировании).

### 6.5 Итоговый Score

```
score = baseScore × fibScore × durationScore × geoScore × fullFitBonus
```

При наличии хвоста и нарушения правил хвоста — `tailPenalty` ∈ [0, 1] умножается отдельно.

## 7. Контекст родительской волны

`WaveContext` несёт информацию о позиции текущей волны в родительской:

```csharp
record WaveContext(
    ElliottModelType ParentModel,
    int WavePosition,          // 1..5 или 1..3
    bool ExpectedIsUp,         // ожидаемое направление
    double MinRetracementRatio,// минимально допустимый откат
    double MaxRetracementRatio // максимально допустимый откат
);
```

### Дополнительные ограничения из контекста

| Условие | Проверка |
|---|---|
| Глубина отката (волна 2 импульса) | `retracementRatio ∈ [minRetracement, maxRetracement]` |
| Разрешённые модели для позиции | `ElliottWavePatternHelper.ModelRules[parentModel].Models[waveKey].Contains(candidateModel)` |
| Направление | `actualIsUp == expectedIsUp` |
| Запрет перекрытия (W4 / W1) | При позиции 4 проверяем не пересекает ли конец W4 начало W1 |

Если контекст нарушен — `contextScore = 0.0` (модель отвергается).
Иначе откат в пределах «типичного» диапазона для данной позиции увеличивает Score
через `fibScore` соответствующей карты коррекции.

## 8. Рекурсивное расширение (v2, не реализовано)

В v2 вместо `SIMPLE_IMPULSE` для каждой подволны будет вызываться рекурсивный
`Parse(subSegments, allowedModels, context)`.

Параметры для рекурсивного вызова:
- `segments` — сегменты, относящиеся к данной подволне.
- `targetModel` — может быть `null` (искать лучшую) или ограничен `allowedModels` из
  `ModelRules`.
- `parentContext` — строится из позиции подволны в текущей модели.

Score дочернего узла умножается на Score родителя.

Глубина рекурсии ограничена `MAX_DEEP_LEVEL = 10`.

## 9. Предложения по реализации в C#

### 9.1 Структура данных

```csharp
// Один результат разметки
record ParsedModel(
    ElliottModelType ModelType,
    BarPoint[] WaveEnds,      // [0]=start, [1..K]=концы подволн
    double Score,
    bool HasTail,
    int TailLength            // число сегментов в хвосте
);
```

### 9.2 Хранение правил проверки модели

```csharp
// На каждую модель — один объект с набором чекеров
class ModelChecker
{
    public ElliottModelType Model { get; }

    // Возвращает null при успехе, иначе — причину провала
    public Func<WaveSpan[], string?> HardRulesCheck { get; }

    // Возвращает скоринговый коэффициент [0..1]
    public Func<WaveSpan[], double> SoftScoreCalc { get; }

    // Правило для хвоста
    public Func<WaveSpan[], Segment, TailRuleResult>? TailRule { get; }
}
```

Все экземпляры `ModelChecker` регистрируются в статическом словаре в файле
`ElliottWaveMarkupRules.cs` (отдельный файл для удобного редактирования правил).

### 9.3 Минимальная сложность поиска сегментов

Ключевой принцип: в v1 каждая подволна = 1 сегмент. Перебор тривиален:

```
для каждой модели M с K подволнами:
    взять первые K сегментов
    проверить hard-правила
    если ок → посчитать Score
    если S > K → проверить хвост
```

Нет необходимости в DP или beam search на одном уровне — линейный проход O(M×K) где
M ≈ 9 (число моделей), K ≤ 5. Вся сложность уйдёт в рекурсию v2.

### 9.4 Выражение геометрических правил

Для треугольников и диагоналей ввести вспомогательную функцию:

```csharp
static double TrendlineScore(BarPoint lineStart, BarPoint lineEnd,
                              BarPoint testPoint, double patternRange,
                              double k = 3.0)
{
    // Проекция testPoint на прямую lineStart→lineEnd
    double expectedValue = Interpolate(lineStart, lineEnd, testPoint.BarIndex);
    double distance = Math.Abs(testPoint.Value - expectedValue);
    return Math.Exp(-k * distance / patternRange);
}
```

Это единственная функция для §20.2–20.5, параметр `k` управляет «жёсткостью».
Для диагоналей с разрешённым throw-over добавляется проверка знака отклонения:
только отклонение **за** линию штрафуется сильнее (другой `k`).

### 9.5 Подсчёт пересечений образующей (диагонали, §20.4–20.5)

```csharp
static int CountTrendlineCrossings(BarPoint[] subWaveEnds, BarPoint lineA, BarPoint lineB)
{
    int crossings = 0;
    for (int i = 0; i < subWaveEnds.Length - 1; i++)
    {
        double prevExpected = Interpolate(lineA, lineB, subWaveEnds[i].BarIndex);
        double currExpected = Interpolate(lineA, lineB, subWaveEnds[i+1].BarIndex);
        bool prevAbove = subWaveEnds[i].Value > prevExpected;
        bool currAbove = subWaveEnds[i+1].Value > currExpected;
        if (prevAbove != currAbove) crossings++;
    }
    return crossings;
}
```

Затем `crossingPenalty = Exp(−0.5 × crossings)`.
