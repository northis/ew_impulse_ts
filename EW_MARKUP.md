## 1. Выбор участка для разметки

Индикатор принимает параметр `N` — глубина в свечах. Алгоритм:

1. Берём последние `N` баров.
2. Находим глобальный **максимум** и **минимум** этого диапазона (два «граничных» экстремума).
3. Определяем, какой из двух граничных экстремумов **ближе к последнему бару** — назовём его `nearExtremum`.
4. Между `nearExtremum` и последним баром ищем **контр-экстремум** (первый экстремум, направленный в противоположную сторону от `nearExtremum`) — назовём его `counterExtremum`.
5. **Участок разметки**: от `nearExtremum` до `counterExtremum`.

> Пример (бычий рынок, последний бар у максимума):
> - Глобальный максимум — ближний к финалу (`nearExtremum`).
> - Между ним и концом — первый локальный минимум (`counterExtremum`).
> - Размечаем участок максимум → минимум как нисходящую волну.

**Что закладывается на будущее (v2):** кроме описанного участка, возможен режим
«разметка до последнего бара» — тогда `counterExtremum` заменяется последним баром.
Логика идентична, просто последний бар используется как незавершённый экстремум.

## 2. Препроцессинг: адаптивный выбор ранга и построение сегментов

Цель — получить набор **элементарных колебаний** (up/down отрезков) подходящей
крупности, из которых алгоритм будет собирать модели.

### 2.1 Адаптивный подбор ранга R

Фиксированный период 0.01 непригоден: он даёт слишком мелкую пилу, в которой
каждая волна модели должна занимать десятки сегментов. Вместо этого:

1. Начинаем с **максимального ранга** `R_max` — минимального `DeviationPercent`,
   при котором `SimpleExtremumFinder` даёт на участке ровно **1 сегмент**
   (от глобального максимума до глобального минимума). Это верхняя граница.
2. Последовательно делим `DeviationPercent` вдвое, записывая число сегментов `S`
   при каждом шаге: `R_max, R_max/2, R_max/4, ...`
3. **Минимальный рабочий ранг** `R` — первый, при котором `S ≥ K_min`,
   где `K_min = min(ExpectedWaves)` среди всех кандидатных моделей (обычно 3).
4. Для анализа используем **диапазон рангов** `[R, R+2]` — три последовательных
   шага деления. Каждый ранг порождает свой набор точек и свои кандидаты.

> Если ни при каком ранге `S < K_min` — участок слишком короткий, разметка невозможна.

### 2.2 Два уровня зигзага на каждом ранге

Для каждого рабочего ранга `Rk` строим **два** набора точек:

- **Основной (ранг Rk):** используется как опорные конечные точки волн.
  Число сегментов `S ∈ [K, 2K]` — желаемый диапазон для данного ранга.
- **Верификационный (ранг Rk+1):** один шаг деления мельче. Используется
  **только** для проверки внутренней корректности найденных волн (§4.4).
  Не участвует в определении позиций волн напрямую.

### 2.3 Структура сегментов

Из точек основного ранга строим **список сегментов**:
каждый сегмент = пара `(start: BarPoint, end: BarPoint)`.

Число сегментов `S = len(segments)`. В v1 каждая подволна модели = ровно 1 сегмент.
Импульс требует не менее 5 сегментов, зигзаг — 3, треугольник — 5 и т.д.

## 3. Главный метод разметки

Публичная точка входа:

```
Parse(
    points           : List<BarPoint>          // экстремумы участка
) → List<ExactParsedNode>                     // отсортировано по убыванию Score
```

Внутренняя рекурсивная реализация:

```
ParseInternal(
    points           : List<BarPoint>,
    allowedModels    : ElliottModelType[]?,    // null → все TargetModels
    depth            : int                     // 0 = верхний уровень
) → List<ExactParsedNode>                     // отсортировано по убыванию Score
```

`Parse(points)` вызывает `ParseInternal(points, null, 0)`. После нахождения
наилучших кандидатов (если `depth + 1 < MAX_MARKUP_DEPTH`) каждый кандидат
передаётся в `FillSubWaveModels` для рекурсивной идентификации подволн (§4.5).
По завершении всех рекурсивных вызовов список **пересортировывается** с учётом
`depthCoverageBonus` (§6.6): кандидаты, чьи подволны полностью опознаны на
заданной глубине, получают приоритет.

### Логика перебора

1. Формируем список кандидатов для проверки:
   - если `targetModel != null` → только эта модель;
   - иначе → `TargetModels` (см. `ElliottWaveExactMarkup.TargetModels`).

2. Для каждой модели из кандидатов:
     - Определяем `K = ExpectedWaves(model)` — требуемое число подволн.
     - Если `S < K` → **пропуск** (не хватает сегментов).
     - Если `S == K` → **«полная посадка»**: берём ровно `K` сегментов
         (сегменты 0..K-1), модель занимает весь доступный диапазон. Такой
         кандидат получает бонус `fullFitBonus` и имеет приоритет выше.
     - Если `S > K` → **«с хвостом»**: берём первые `K` сегментов, начиная с
         позиции 0 (т.е. модель занимает сегменты 0..K-1, сегменты K..S-1
         образуют хвост). В v1 всегда только одна стартовая позиция — 0.
     - **§4.1-continuity (hard rule):** модель обязана охватывать весь входной
         диапазон от `altPoints[0]` до `altPoints[n-1]` — сдвинутые окна
         (offset > 0, не начинающиеся с `altPoints[0]`) **запрещены**.
         Это гарантирует, что ни один экстремум входного ряда не будет
         «пропущен» и проигнорирован (см. §4.1-continuity ниже).
     - **Дополнительный проход для усечения** (только для `IMPULSE` и
         `DIAGONAL_CONTRACTING_ENDING`, см. §5.2):
         - Если `S ≥ K` — пробуем взять первые `K-1` сегментов как W1..W4,
             а `segments[K-1]` (следующий сразу после W4) — как кандидат на усечённую W5.
         - Проверяем условия §5.2; при успехе добавляем кандидат с `ParseStrategy.TruncatedWave5`.

3. Для каждого кандидата считаем **Score** (см. §6).

4. Проверяем `parentContext` — дополнительные ограничения (§7).

5. Добавляем в результирующий список `ParsedModel` с рассчитанным Score.

6. Сортируем по убыванию Score и возвращаем.

7. После завершения `FillSubWaveModels` для всех кандидатов — применяем
   `depthCoverageBonus` (§6.6) к каждому Score и **пересортируем** список.

## 4. Поиск модели в цепочке сегментов

### 4.1 Проверка базовых ценовых правил

Для каждого кандидата с `K` сегментами проверяем hard-правила (нарушение → Score = 0,
модель исключается):

| Модель | Ключевые hard-правила |
|---|---|
| IMPULSE | W3 ≥ max(W1, W5); W4 не пересекает территорию W1; ни одна свеча в любой подволне не пробивает начало импульса (для восходящего: `Low[i] ≥ start`, для нисходящего: `High[i] ≤ start`) |
| DIAGONAL (сходящийся) | каждая волна короче предыдущей одноимённой; W4 пересекает W1; ни одна свеча не пробивает начало диагонали |
| DIAGONAL (расходящийся) | каждая волна длиннее предыдущей; ни одна свеча не пробивает начало диагонали |
| ZIGZAG | C ≥ 0.618 × A; B ≤ 0.95 × A; ни одна свеча в любой подволне не пробивает начало паттерна |

> **Правило §4.1-candle (`CheckCandleBreachRules`):** проверяется на уровне OHLC-баров.
> Применяется к моделям: IMPULSE, DIAGONAL_CONTRACTING_INITIAL, DIAGONAL_CONTRACTING_ENDING, ZIGZAG, DOUBLE_ZIGZAG.
> Восходящий паттерн: `Low[i] ≥ startPrice` для каждого бара в каждой подволне.
> Нисходящий паттерн: `High[i] ≤ startPrice`.
> Активна только при наличии реального `IBarsProvider`; в unit-тестах (без провайдера) пропускается.
> **Правило §4.1-continuity (`FullSpanOnlyRule`) — hard:** модели на любом уровне разметки обязаны покрывать **весь** входной диапазон без пропусков.
> Конец подволны `i` должен совпадать (по `BarIndex`) с началом подволны `i+1`.
> На уровне отбора кандидатов это означает: разрешены только **полно-охватывающие** комбинации,
> стартующие в `altPoints[0]` и заканчивающиеся в `altPoints[n-1]`. Частичные окна (offset > 0),
> пропускающие ведущие или замыкающие экстремумы, отбрасываются как нарушающие непрерывность.
> На уровне подволн (`FillSubWaveModels`) принимается только тот результат, у которого
> `StartPoint.BarIndex == fromBar` и `EndPoint.BarIndex == toBar`;
> результаты, не охватывающие весь диапазон подволны, отвергаются.
> **Нарушение → HARD: кандидат не добавляется в список результатов.**
> **Правило §4.1-endpoint (`CheckEndpointExtremum`) — hard:** проверяется **только на последнем уровне разметки** (depth + 1 ≥ MAX_MARKUP_DEPTH).
> На последнем уровне мы не можем разобрать внутреннюю структуру волны, поэтому обязательное условие:
> начало и конец сегмента должны быть абсолютными экстремумами цены внутри всего барового диапазона модели.
> Применяется к моделям: IMPULSE, DIAGONAL_CONTRACTING_INITIAL, DIAGONAL_CONTRACTING_ENDING, ZIGZAG, DOUBLE_ZIGZAG.
> Восходящий паттерн: `High[i] ≤ endPrice` для каждого бара (endPrice есть максимальный High).
> Нисходящий паттерн: `Low[i] ≥ endPrice` (endPrice есть минимальный Low).
> Начальная сторона (старт = экстремум) уже обеспечена правилом §4.1-candle.
> **Нарушение → HARD: родительский кандидат полностью отбрасывается** (`FillSubWaveModels` возвращает `false`).
> Исключения: усечённые волны 5 (по правилам допустимы) и флет в волнах B/X — маловероятные случаи,
> требующие внутренней разметки; на последнем уровне они неверифицируемы и отбрасываются.
> Активна только при наличии реального `IBarsProvider`.
| FLAT_EXTENDED | B ≥ A (выходит за начало); C ≥ 1.618 × A |
| FLAT_RUNNING | B ≥ A; C ≤ 1.618 × A |
| TRIANGLE_CONTRACTING | каждая волна короче предыдущей |
| TRIANGLE_RUNNING | то же + B превышает начало треугольника |

### 4.2 Допустимые sub-модели

Каждая подволна в v1 изначально представлена одним сегментом основного ранга.
После базовой разметки алгоритм рекурсивно идентифицирует под-модель каждой
подволны (см. §4.6). Допустимые под-модели для каждой позиции определяются
правилом `ElliottWavePatternHelper.ModelRules[parentModel].Models[waveKey]`.

### 4.3 Идентификация «расширенной» волны / ведущей волны

Для импульса — дополнительно выясняем:
- W3 / W1 > 1.0 → W3 расширенная (стандартный случай).
- W1 > W3 (W1 расширенная) → допустимо, но более редко.
- W5 > W3 → W5 расширенная.

Это влияет на скоринг по картам Фибоначчи, но не является hard-ограничением.

### 4.4 Внутренняя валидация через верификационный ранг

После прохождения hard-правил по основному рангу проверяем точки **верификационного
ранга (Rk+1)** — более мелкого зигзага — на предмет нарушений внутри каждой подволны.
Правила зависят от **позиции волны в модели**:

| Позиция | Ожидаемое поведение | Действие при нарушении |
|---|---|---|
| Импульсные волны (1, 3, 5 в IMPULSE; A и C в ZIGZAG) | Нет выхода за границы сегмента | HARD: отбросить кандидата |
| Глубокая коррекция (W2, W4 как ZIGZAG) | Нет выхода за границы | HARD: отбросить |
| Flat B (FLAT_EXTENDED / FLAT_RUNNING) | **Обязан** выйти за начало A | HARD: отбросить если нет вылета |
| Flat B (FLAT_RUNNING) | Вылет за начало паттерна (B > A по абсолюту) | Степень вылета → soft-score |
| Flat B (FLAT_REGULAR) | Не должен выйти за начало A; желательно завершение вблизи origin | HARD: отбросить при вылете за origin; близость к origin → soft-score |
| Triangle B (RUNNING) | Выходит за начало треугольника | HARD: отбросить если нет вылета |
| Triangle C, D, E | Небольшие выходы допустимы | Soft: штраф пропорционально амплитуде вылета |

**Порог «допустимого вылета»** для мягких позиций:
```
overshoot = |innerExtremum - waveEnd| / waveLength
overshootPenalty = Exp(-3 × overshoot)   // мультипликативный в Score
```
Где `innerExtremum` — самый дальний экстремум верификационного ранга внутри данного
сегмента, выходящий за его границы.

### 4.5 Рекурсивная идентификация подволн

После построения кандидата верхнего уровня алгоритм рекурсивно обходит каждую
подволну и пытается определить её модель (`FillSubWaveModels`).

**Константа глубины:** `MAX_MARKUP_DEPTH = 3`.
- Глубина 0 — верхний уровень (сам паттерн).
- Глубина 1 — подволны паттерна (первый вложенный уровень).
- Глубина 2 — подволны подволн (второй вложенный уровень).
- На глубине `MAX_MARKUP_DEPTH` рекурсия останавливается; подволны остаются
  как `SIMPLE_IMPULSE`.

**Допустимые под-модели** для каждой позиции берутся из
`ElliottWavePatternHelper.ModelRules[parentModel].Models[waveKey]`
(метод `GetValidSubModelsForWave`).

**Пример для ZIGZAG:**

| Волна | Допустимые под-модели |
|---|---|
| A | IMPULSE, DIAGONAL_CONTRACTING_INITIAL, DIAGONAL_EXPANDING_INITIAL |
| B | ZIGZAG, DOUBLE_ZIGZAG, TRIPLE_ZIGZAG, FLAT_EXTENDED, FLAT_RUNNING, FLAT_REGULAR, TRIANGLE_CONTRACTING, TRIANGLE_RUNNING, TRIANGLE_EXPANDING |
| C | IMPULSE, DIAGONAL_CONTRACTING_ENDING, DIAGONAL_EXPANDING_ENDING |

**Алгоритм `FillSubWaveModels`:**
1. Для каждой подволны `i` в `node.SubWaves`:
   - Определить `waveKey = GetWaveKey(parentModel, i+1)`.
   - Получить `validModels = GetValidSubModelsForWave(parentModel, waveKey)`.
   - Собрать все точки из входного списка в диапазоне `[fromBar, toBar]` сегмента;
     убедиться, что концевые точки сегмента присутствуют.
   - Вызвать `ParseInternal(subPoints, validModels, depth)`.
   - **§4.1-continuity (hard):** принять только результат, у которого
     `StartPoint.BarIndex == fromBar` и `EndPoint.BarIndex == toBar`.
     Результаты, не охватывающие весь диапазон подволны, отвергаются.
   - Заменить `node.SubWaves[i]` лучшим результатом с `WaveCount == ExpectedWaves`
     и точным охватом диапазона.
   - **Проверка §4.1-endpoint** — **HARD** (only if `depth + 1 ≥ MAX_MARKUP_DEPTH` and all `validModels`
     are motive/zigzag): if the sub-wave is still `SIMPLE_IMPULSE` (no satisfactory sub-model
     found) and the segment violates the endpoint-extremum rule, `FillSubWaveModels` returns
     `false` — the **entire parent candidate is discarded**.

### 4.6 Light-валидация для флетов и треугольников

Для паттернов со структурным «вылетом» (flat B, triangle running B) применяется
дополнительная проверка через верификационный ранг:

**FLAT_EXTENDED / FLAT_RUNNING:**
1. На верификационном ранге ищем точку, выходящую за уровень начала паттерна
    (волна B должна превысить начало). Берём экстремум с **наибольшим отклонением**
    от начала (максимум при восходящем флете, минимум при нисходящем).
2. Этот экстремум — кандидат на «конец B». Точка непосредственно перед ним
    (в направлении A) — кандидат на «конец A».
3. Если на верификационном ранге между 0 и кандидатной B несколько экстремумов —
    возможна ситуация «b of B»: ближайший экстремум является подволной внутри B,
    а не концом A. В таком случае порождаем **два кандидата** `ParsedModel`:
    - `ParseStrategy.PrimaryRank` — разбиение по основному рангу.
    - `ParseStrategy.AnchorShifted` — разбиение с альтернативной позицией A/B.
    Оба кандидата скорируются независимо; Fibonacci-score выберет лучший.

**FLAT_REGULAR:**
1. На верификационном ранге ищем экстремум волны B, который подходит максимально
    близко к origin, но **не пересекает** его.
2. Если найден вылет за origin — кандидат отбрасывается.
3. Чем ближе конец B к origin без пробоя, тем выше soft-score для regular flat.

**TRIANGLE_CONTRACTING / TRIANGLE_RUNNING:**
1. Ищем конец волны A как первый крупный экстремум на основном ранге.
2. Проверяем, что конец волны E попадает в диапазон `(начало треугольника, конец A)`.
3. Требуем минимальное число баров между B и E:
   `barCount(C..E) ≥ barCount(A) × 0.3` — защита от «карликовых» C, D, E.
   Нарушение → soft-penalty, не hard-reject.
4. Направление E должно совпадать с направлением A (обе — в сторону коррекции).

**Глубина коррекции треугольника** (`correctionDepth`) проверяется только
при наличии `parentContext` (§7) — когда треугольник является подволной старшей
модели. Без контекста это правило **не применяется**: скорируются только
отношения волн по картам Фибоначчи.
```
// только если parentContext != null:
correctionDepth = |end(E) - start(triangle)| / |wave_before_triangle|
assert correctionDepth ∈ [0.10, 0.950]   // для TRIANGLE_RUNNING нижняя граница 0.10
```

## 5. Правила «хвоста»

Если модель занимает только первые `K` из `S` сегментов, остаток (`tail`) может
аннулировать найденную модель или снизить её Score.

**Ключевой принцип:** хвост проверяется на **противоречие** найденной модели.
Если хвост структурно несовместим с тем, что обычно следует за данной моделью —
модель считается невалидной в данных границах и отбрасывается целиком. Если
противоречия нет, но хвост «неудобен» — снижаем Score.

### 5.1 Правила отмены по хвосту

| Модель | Условие отмены (HARD) | Условие снижения Score (SOFT) |
|---|---|---|
| TRIANGLE_CONTRACTING | Первый сегмент хвоста идёт **против** ожидаемого thrust-а (выброс из треугольника всегда в сторону волны, противоположной E) | Хвост слишком короткий для thrust — слабый выброс |
| TRIANGLE_RUNNING | То же | То же |
| IMPULSE (восходящий) | Первый сегмент хвоста уходит **ниже** конца волны 4 — это означает, что «волна 5» импульса ещё не завершена; либо вся структура неверна | Хвост глубже конца волны 2, но не ниже волны 4 — умеренный штраф |
| IMPULSE (нисходящий) | Первый сегмент хвоста уходит **выше** конца волны 4 | Аналогично |
| DIAGONAL_CONTRACTING_ENDING | Первый сегмент хвоста идёт **по** направлению диагонали (нет sharp reversal — цена продолжает движение в сторону диагонали) | Sharp reversal есть, но слабый — штраф |
| ZIGZAG / FLAT | Нет условий отмены | Отсутствие завершённого следующего движения снижает Score |

### 5.2 Усечённая волна 5 (truncation)

По правилам EW_RULES.md (каталог моделей и описания `IMPULSE` /
`DIAGONAL_CONTRACTING_ENDING`) усечение допустимо с вероятностью 5% для
`IMPULSE` и `DIAGONAL_CONTRACTING_ENDING`.

**Ситуация:** алгоритм взял первые `K-1` сегментов как W1..W4 (отдельный проход
из §3 пункт 2d); `segments[K-1]` — кандидат на усечённую W5 (первый сегмент
сразу после W4).

**Алгоритм обнаружения усечённой W5:**

1. Взять `K-1` сегментов как основную часть модели (W1..W4 для IMPULSE).
2. Проверить hard-правила для W1..W4 (стандартная проверка по §4.1, кроме W5).
3. `candidate = segments[K-1]` — следующий сегмент.
4. Проверяем: `candidate` идёт **в направлении тренда** (та же сторона, что W1 и W3).
5. Длина `candidate` < длины W3 по абсолютному значению (главное условие
   усечения: W5 не превышает W3).
6. Длина `candidate` ≥ 0.1 × W1 (не слишком карликовая волна).
7. Если всё выполнено — создаём дополнительный кандидат `ParsedModel` с
   `ParseStrategy.TruncatedWave5`, в котором `candidate` включается как W5,
   а Score умножается на коэффициент усечения `0.05` (базовая вероятность).

**Для `DIAGONAL_CONTRACTING_ENDING`** логика аналогична, но:
- W5 диагонали может быть усечена (не должна превышать конец W3).
- Дополнительно проверяем сходимость: конец усечённой W5 должен быть
  **ближе к каналу 1–3**, чем конец W4 к каналу 2–4.

```
// Псевдокод обнаружения truncation (вызывается из §3, п. 2d)
if (segments.Count >= K)   // K-1 основных + кандидат на W5
{
    WaveSpan[] wavesK1 = BuildWaves(segments, count: K - 1);
    if (HardRulesCheck(model, wavesK1) == null)   // W1..W4 валидны
    {
        Segment candidate = segments[K - 1];
        bool inTrendDirection = (candidate.IsUp == modelIsUp);
        bool shorterThanW3 = candidate.Length < wavesK1[2].Length;
        bool notTooSmall = candidate.Length >= wavesK1[0].Length * 0.1;

        if (inTrendDirection && shorterThanW3 && notTooSmall)
        {
            yield return CloneWithTruncatedW5(model, wavesK1, candidate,
                             scoreMultiplier: 0.05);
        }
    }
}
```

### 5.3 Реализация правил хвоста в C#

```csharp
static readonly Dictionary<ElliottModelType, Func<WaveSpan[], Segment[], TailRuleResult>> TailRules =
new()
{
    [ElliottModelType.TRIANGLE_CONTRACTING] = (waves, tail) =>
    {
        if (tail.Length == 0) return TailRuleResult.Valid;
        // Ожидаемый thrust = против направления волны E (последней)
        bool expectedThrustUp = !waves[4].IsUp;
        bool actualUp = tail[0].IsUp;
        return actualUp == expectedThrustUp
            ? TailRuleResult.Valid
            : TailRuleResult.Rejected;   // противоречит модели
    },
    [ElliottModelType.IMPULSE] = (waves, tail) =>
    {
        if (tail.Length == 0) return TailRuleResult.Valid;
        bool isUp = waves[0].IsUp;
        double w4End = waves[3].EndValue;
        double tailEnd = tail[0].EndValue;
        // Хвост уходит за конец W4 в сторону против тренда — модель невалидна
        bool violated = isUp ? tailEnd < w4End : tailEnd > w4End;
        return violated ? TailRuleResult.Rejected : TailRuleResult.Valid;
    },
    // ...
};
```

## 6. Скоринг

Raw score — вещественное число `(0, +∞)`, произведение множества компонентов.
После расчёта все raw score в результирующем списке **нормируются** к диапазону
`[0, 1]` делением на максимальный score в списке.

### 6.1 Базовая вероятность модели

Из таблицы §2 EW_RULES.md: `modelCoeff / 1.0` (нормализуется к 1.0 как максимуму).

```
baseScore = modelProbabilityCoeff / MaxModelCoeff   // MaxModelCoeff = 1.0
```

### 6.2 Соответствие уровням Фибоначчи

Для каждой пары (волна i+1 / волна i) вычисляем реальное отношение длин и сравниваем с
ближайшим уровнем в соответствующей карте Фибоначчи (§4 EW_RULES.md).

```
fibScore_i = weight(nearestFibLevel) / sum(allWeights)
           × exp(−k × |actualRatio − nearestFibLevel| / nearestFibLevel)
```

где `k ≈ 5` управляет «жёсткостью» штрафа за отклонение от уровня.

Общий `fibScore = ∏ fibScore_i`.

### 6.3 Соразмерность продолжительности волн (§20 EW_RULES.md)

Для импульса (§20.1):

```
durationScore = durationPenalty(W2, W4, limit_4over2=1.5, limit_2over4=1.1)
```

Для треугольников и диагоналей — геометрический score (§20.2–20.5):
```
geoScore = exp(−k × distanceToLine / patternRange)
```

### 6.4 Бонус «полной посадки»

Если модель заняла все `S` сегментов без хвоста:
```
fullFitBonus = 1.2
```
`fullFitBonus > 1.0` допускает raw score > 1.0. Это нормально: до финального вывода
score трактуется как **относительный ранг**, а не как вероятность.

### 6.5 Итоговый Score (до рекурсии)

```
rawScore = baseScore × fibScore × durationScore × geoScore × fullFitBonus
```

При наличии хвоста и нарушения правил хвоста — `tailPenalty` ∈ [0, 1] умножается отдельно.

```
finalScore = rawScore / max(rawScore over all accepted candidates)
```

### 6.6 Бонус глубины разметки (`depthCoverageBonus`)

После того как `FillSubWaveModels` рекурсивно заполнила подволны, у каждого
кандидата вычисляется **коэффициент охвата глубины** — доля подволн (и их
подволн), опознанных как реальная модель (не `SIMPLE_IMPULSE`):

```
ComputeDepthCoverage(node):
    если node не имеет подволн → вернуть 1.0  // лист, полностью определён
    sum = 0
    для каждой подволны sw:
        если sw.ModelType != SIMPLE_IMPULSE:
            sum += ComputeDepthCoverage(sw)     // рекурсия
        // SIMPLE_IMPULSE вносит 0
    вернуть sum / WaveCount                     // среднее по подволнам
```

Значение `depthCoverage ∈ [0, 1]`: 1.0 — все подволны опознаны на всех уровнях.

Константа `DEPTH_COVERAGE_BONUS = 0.5`. Бонус умножается на Score **после** рекурсии:

```
adjustedScore = rawScore × (1 + depthCoverage × DEPTH_COVERAGE_BONUS)
```

Максимальный бонус — `×1.5` при полном охвате. Минимум — `×1.0` (без изменений).

После применения бонуса список кандидатов **пересортировывается** по убыванию
`adjustedScore`. Таким образом, при равных фибоначчи-коэффициентах приоритет
получает тот кандидат, чья разметка достигла максимальной глубины `MAX_MARKUP_DEPTH`.

> Бонус применяется только на уровнях, где выполнялась рекурсия
> (`depth + 1 < MAX_MARKUP_DEPTH`). На последнем уровне рекурсия отсутствует
> и пересортировка не нужна.

### 6.7 Итоговый Score (после рекурсии)

```
rawScore      = baseScore × fibScore × durationScore × geoScore × fullFitBonus
              × tailPenalty          // [0, 1]; 1.0 если хвост не нарушает правил
adjustedScore = rawScore × (1 + depthCoverage × DEPTH_COVERAGE_BONUS)
finalScore    = adjustedScore / max(adjustedScore over all accepted candidates)
```

## 7. Контекст родительской волны

`WaveContext` несёт информацию о позиции текущей волны в родительской:

```csharp
record WaveContext(
    ElliottModelType ParentModel,
    int WavePosition,          // 1..5 или 1..3
    bool ExpectedIsUp,         // ожидаемое направление
    double MinRetracementRatio,// минимально допустимый откат
```

### Дополнительные ограничения из контекста

| Условие | Проверка |
|---|---|
| Глубина отката (волна 2 импульса) | `retracementRatio ∈ [minRetracement, maxRetracement]` |
| Разрешённые модели для позиции | `ElliottWavePatternHelper.ModelRules[parentModel].Models[waveKey].Contains(candidateModel)` |
| Направление | `actualIsUp == expectedIsUp` |
| Запрет перекрытия (W4 / W1) | При позиции 4 проверяем не пересекает ли конец W4 начало W1 |

Иначе откат в пределах «типичного» диапазона для данной позиции увеличивает Score
через `fibScore` соответствующей карты коррекции.

## 8. Рекурсивное расширение (реализовано в v1, глубина 3)

`MAX_MARKUP_DEPTH = 3` определяет три уровня идентификации:
- **Уровень 0** — главная модель по всему диапазону.
- **Уровень 1** — под-модели каждой подволны главной модели.
- **Уровень 2** — под-модели подволн первого уровня.

Алгоритм описан в §4.5. Точка входа — `Parse(points)`, которая вызывает
`ParseInternal(points, null, 0)`. После успешного нахождения кандидата
`FillSubWaveModels` рекурсивно запускает `ParseInternal` для каждой подволны
с ограниченным набором `allowedModels`, взятым из `ElliottWavePatternHelper.ModelRules`.

**Отображение в индикаторе** использует нотацию двух уровней:
- Уровень 0 (Minuette, `case 4`): `(i) (ii) (iii) (iv) (v)` / `(a) (b) (c)`
- Уровень 1 (Subminuette, `case 3`): `i ii iii iv v` / `a b c`

Метод `DrawMarkupNode(node, color, prefix, notationLevel)` рекурсивно обходит
дерево `ExactParsedNode` и рисует метку и линию для каждой подволны. При
`notationLevel > 0` и под-модели, отличной от `SIMPLE_IMPULSE`, выполняется
рекурсивный вызов для отображения вложенных подволн.

### 9.3 Сложность поиска с адаптивным рангом

Ключевой принцип v1: каждая подволна = 1 сегмент основного ранга.

```
1. Найти R_max (1 сегмент на участке)
2. Для каждого ранга Rk ∈ {R, R+1, R+2}  // три шага: DeviationPercent = R, R/2, R/4
   пока S ∈ [K_min, 2·K_max]:
   для каждой модели M с K подволнами:
       если S < K → пропуск
       взять первые K сегментов
       проверить hard-правила (§4.1)
       если ок:
           запустить inner-validation по верификационному рангу Rk+1 (§4.4)
           если паттерн со структурным вылетом → породить AnchorShifted кандидата (§4.5)
           посчитать Score
           если S > K → проверить хвост
```

Сложность: `O(logRange × M × K × S_fine)` где:
- `logRange ≈ 3` (три ранга)
- `M ≈ 9` (число моделей)
- `K ≤ 5` (подволны)
- `S_fine` — число точек на верификационном ранге, обычно `2K..4K`

Это **полиномиально**. Вся экспоненциальная сложность уйдёт в рекурсию v2.

### 9.4 Выражение геометрических правил

Для треугольников и диагоналей ввести вспомогательную функцию:

```csharp
static double TrendlineScore(BarPoint lineStart, BarPoint lineEnd,
                              BarPoint testPoint, double patternRange,
                              double k = 3.0)
{
    // Проекция testPoint на прямую lineStart→lineEnd
    double expectedValue = Interpolate(lineStart, lineEnd, testPoint.BarIndex);
    double distance = Math.Abs(testPoint.Value - expectedValue);
    return Math.Exp(-k * distance / patternRange);
}
```

Это единственная функция для §20.2–20.5, параметр `k` управляет «жёсткостью».
Для диагоналей с разрешённым throw-over добавляется проверка знака отклонения:
только отклонение **за** линию штрафуется сильнее (другой `k`).

### 9.5 Подсчёт пересечений образующей (диагонали, §20.4–20.5)

```csharp
static int CountTrendlineCrossings(BarPoint[] subWaveEnds, BarPoint lineA, BarPoint lineB)
{
    int crossings = 0;
    for (int i = 0; i < subWaveEnds.Length - 1; i++)
    {
        double prevExpected = Interpolate(lineA, lineB, subWaveEnds[i].BarIndex);
        double currExpected = Interpolate(lineA, lineB, subWaveEnds[i+1].BarIndex);
        bool prevAbove = subWaveEnds[i].Value > prevExpected;
        bool currAbove = subWaveEnds[i+1].Value > currExpected;
        if (prevAbove != currAbove) crossings++;
    }
    return crossings;
}
```

Затем `crossingPenalty = Exp(−0.5 × crossings)`.

## 10. Стратегия тестирования

### 10.1 Общий подход

Тесты используют `PatternGenerator` для синтетической генерации паттернов и
`ElliottWaveExactMarkup.Parse` для их разметки. Сгенерированные ключевые точки
(`PatternKeyPoints` с уровнем `model.Level`) подаются напрямую в `Parse` как
«идеальные» экстремумы. Это позволяет проверить корректность оценки и выбора
наилучшей модели в контролируемых условиях.

### 10.2 Требования к генерации

- **Минимум 200 баров** на паттерн. При `BarsCount < SIMPLE_BARS_THRESHOLD (100)`
  генератор подставляет плоские `SIMPLE_IMPULSE`-подволны вместо реальных субволн.
  При 200 барах генерируются полноценные рекурсивные субструктуры. Для треугольников и
  диагоналей (5 подволн) рекомендуется ≥ 300 баров, чтобы каждая подволна получала
  достаточно места.

- **Повторные прогоны** (по умолчанию 20 итераций на тип модели). Каждая итерация
  независима: случайный seed `PatternGenerator` создаёт разные паттерны, что устойчиво
  проверяет алгоритм. Порог успеха: ≥ 60 % итераций правильно идентифицируют тип.

### 10.3 Построение входных точек для `Parse`

1. Берём `model.PatternKeyPoints` — словарь `DateTime → List<PatternKeyPoint>`.
2. Отбираем точки с `Notation.Level == model.Level` — только главные подволны.
3. Находим первый экстремум паттерна как точку до индекса первой ключевой точки.
4. Добавляем начальную точку `(startVal, startDate, barsProvider)`.
5. Добавляем все ключевые точки уровня `model.Level` в `List<BarPoint>` по индексу.
6. Дедуплицируем по `BarIndex` (сохраняем точку с максимальным `Level`).

### 10.4 Критерии прохождения теста

| Условие | Смысл |
|---|---|
| `bestResult.ModelType == expectedModelType` | Разметчик поставил верный тип |
| `bestResult.WaveCount == bestResult.ExpectedWaves` | Паттерн полный (без хвоста) |
| `matchedTests / totalTests >= 0.6` | Минимальный порог точности (60 %) |

### 10.5 Покрытие моделей

Тесты охватывают все типы из `ElliottWaveExactMarkup.TargetModels`:

| `ZigzagExactMarkupTest` | `ZIGZAG` | 3 |
| `DoubleZigzagExactMarkupTest` | `DOUBLE_ZIGZAG` | 3 |
| `FlatExtendedExactMarkupTest` | `FLAT_EXTENDED` | 3 |

| `TriangleContractingExactMarkupTest` | `TRIANGLE_CONTRACTING` | 5 |
