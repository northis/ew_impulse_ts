# EW_R_TRIANGLE — Бегущий треугольник (Running Triangle) как торговый сетап

> Проектный документ. Фиксирует видение отдельного, **инкрементального**
> детектора бегущих треугольников (`TRIANGLE_RUNNING`) для торговли по
> направлению тренда, который этот треугольник корректирует.
>
> Статус: **дизайн, до реализации.** Здесь описан целевой алгоритм и то, чем он
> отличается от текущего [`TriangleSetupFinder`](TradeKit.Core/ElliottWave/TriangleSetupFinder.cs).
> Реализацию начинаем после ревью этого документа (см. §11, открытые вопросы).

---

## 1. Мотивация и суть

Бегущий треугольник отрабатывает с большей вероятностью, чем обычный
сжимающийся, потому что его геометрия сама по себе несёт информацию о сильном
тренде: волна **B пробивает начало треугольника (точку 0)**, а значит рынок
«не смог» откатиться и остаётся заряженным в сторону тренда. Поэтому мы делаем
упор именно на этот тип модели и торгуем **тягу (thrust) в направлении тренда**.

Ключевые требования, которые отличают этот детектор от общего разбора EW:

1. **Обязателен предшествующий тренд** достаточной длины, который треугольник
   корректирует. Достаточным считаем тренд, в который укладывается **как минимум
   длина волны B** (см. §5.1 и скриншот в задаче). Это отсекает ложные варианты,
   где то, что мы приняли за волну A треугольника, на самом деле волна W двойного
   зигзага, а «треугольник» — это его волна X (которая улетит в другую сторону и
   нам не подходит).
2. **Running-свойство**: волна B выходит за точку 0. Именно из-за этого свойства
   мы можем спутать волну **C** треугольника с волной **W** — поэтому сильный
   тренд до точки 0 остаётся необходимым условием даже при подтверждённой B.
3. **Волна A не должна вылетать за пределы корректируемого тренда** (тест
   «линии назад», §5.3 — геометрически аналог проверки «заходности» импульса).

## 2. Соотнесение с текущим `TriangleSetupFinder`

Текущий детектор и целевой решают одну задачу принципиально по-разному.

| Аспект | Текущий `TriangleSetupFinder` | Целевой Running-детектор |
|---|---|---|
| Модель времени | **Stateless.** Каждый бар заново сканирует назад по последним экстремумам и собирает ABCDE «с нуля». | **Stateful.** Кандидаты живут между барами, мутируют по мере прихода сегментов зигзага. |
| Тип модели | И `TRIANGLE_CONTRACTING`, и `TRIANGLE_RUNNING` (что подошло). | Только `TRIANGLE_RUNNING`. |
| Валидация цены | Общий код [`ElliottWaveExactMarkupV2.CheckPriceRules`](TradeKit.Core/AlgoBase/) через `TryValidateTriangleV2`. | Собственные явные правила running-геометрии (§5). Общий markup **не используем**. |
| Требование тренда | `IsTrendRatioEnough` — прямое смещение цены ≥ `TRIANGLE_TREND_RATIO`×range за окно перед точкой 0. | **Усиление**: продвижение тренда до точки 0 ≥ **длины волны B** (§5.1). Способ измерения непринципиален. |
| Containment волны A | Удалён (гейт `IsInitialMovement(point0→A)` убивал ~80% на всех треугольниках). | **Возвращаем**, но уже осмысленно — только для running, где A обязана быть начальной коррекцией внутри тренда (§5.3). |
| Тейк-профит | `point0` (начало треугольника). | **Настраиваемый; по умолчанию — конец волны B** (running → тяга уходит дальше точки 0). |
| Стоп-лосс | За волной A. | За волной A (без изменений). |
| Повторные сигналы | Нет (stateless, один проход). | **Настраиваемо**: при росте треугольника вбок можно выдавать новые сигналы (§6.1). |
| Дедупликация | По TP/SL-подписи (`m_ProcessedSignals`). | **По точке 0** (§7.2) — с учётом повторных сигналов при росте вбок. |
| Мультимасштаб | Геометрическая лестница периодов `LADDER_RATIOS`. | Один базовый зигзаг для ABCDE. Старший период не требуется. |

**Вывод (решение O-3).** Логику выносим в **отдельный** `SetupFinder`
(имя `RunningTriangleSetupFinder`), а не переписываем текущий: у них
несовместимая модель состояния. Существующий `TriangleSetupFinder`
**оставляем как есть** (общий поиск треугольников).

Переиспользуем из инфраструктуры:

- [`DeviationExtremumFinder`](TradeKit.Core/Indicators/DeviationExtremumFinder.cs) —
  источник неизменяемых зигзаг-экстремумов (прошлое не меняется — основа
  инкрементальности).
- [`AutoPeriodEstimator.EstimateTrianglePeriod`](TradeKit.Core/AlgoBase/AutoPeriodEstimator.cs)
  для `Period == 0`.
- [`IsInitialMovement`](TradeKit.Core/Common/SingleSetupFinder.cs) —
  как готовая реализация теста containment волны A (§5.3).
- `ElliottWaveSignalEventArgs`, `OnEnterInvoke`/`OnTakeProfitInvoke`/`OnStopLossInvoke`.

## 3. Геометрия и обозначения

Точки (по возрастанию времени): **0, A, B, C, D, E**. Между соседними точками —
один сегмент зигзага (чередование направлений).

Пример для **бычьей тяги** (`isUp`, покупка; медвежий случай зеркален):

```
                        B (выше точки 0 — running)
                       /\              D
   тренд ↑            /  \            /\        ← тяга (thrust) ↑
  ─────────►  0      /    \    C     /  \      /   TP = конец B
             /\     /      \   /\   /    \    /
            /  \   /        \ /  \ /      \  /
    T_start     \ /          C    D        E
                 A                          ↑ E откатывает ≥ 0.5·|D|
                 ↑ SL за волной A
```

- **0 (точка 0)** — конец тренда и начало треугольника (для `isUp` — вершина/high).
- **A** — первая коррекция к тренду (вниз для `isUp`).
- **B** — выходит **за точку 0** (вверх для `isUp`) — *running-свойство*.
- **C** — не пробивает конец A, но возвращается за точку 0 (обратно вниз мимо 0).
- **D** — не пробивает конец B.
- **E** — последний сегмент зигзага, заканчивается где-то между C и D;
  требуем откат ≥ 50% амплитуды D (§6).

Соглашение о направлении наследуем от текущего кода:
`isUp` = тяга вверх = покупка; TP выше SL.

## 4. Инварианты running-геометрии (hard rules)

Для `isUp` (медвежий случай зеркально меняет `>`/`<` и high/low). Все точки —
экстремумы зигзага, `V(x)` = цена точки x.

- **R-TREND** — тренд до точки 0 длиной ≥ |волна B| (§5.1). Проверяется, когда B известна.
- **R-A-INIT** — волна A содержится в тренде (тест «линии назад», §5.3).
- **R-B-RUN** — `V(B) > V(0)` (running: B пробивает точку 0).
- **R-C-0** — `V(C) < V(0)` (C возвращается за точку 0) **и** `V(C) > V(A)`
  (C не пробивает конец A).
- **R-D-B** — `V(D) < V(B)` (D не пробивает B) **и** `V(D) > V(C)`.
- **R-E** — `V(A) < V(E) < V(D)` (E заканчивается между A и D) **и**
  откат E ≥ 0.5·|D| (§6).
- **R-CONTAIN** — внутри каждой волны цена не выходит за её же концы
  (инвариант I2; interior-бары), как в текущем `IsWaveContained`.
- **R-CONVERGE** — сходимость одноимённых концов: `V(C) > V(A)` уже покрывает
  моторную сторону; на корректирующей — `V(D) < V(B)`. (Для running допускаем
  амплитуду |C| > |A|, но конец C не заходит за конец A.)

> Правила R-C-0/R-D-B/R-E формулируем явно вместо `CheckPriceRules`, чтобы
> детектор не зависел от общего markup и мог работать инкрементально
> (проверять по одному новому сегменту зигзага).

## 5. Условие сильного тренда и containment волны A

### 5.1. Тренд до точки 0 (R-TREND)

Тренд, который треугольник корректирует, должен быть **не короче длины
волны B**. То есть продвижение цены в сторону тяги на подходе к точке 0
должно составлять не менее `|V(B) − V(A)|`:

```
продвижение_тренда(до точки 0)  ≥  |V(B) − V(A)|          // длина волны B
```

Геометрия (см. скриншот в задаче): откладываем от точки 0 назад против
тяги отрезок длиной с волну B; тренд должен дотягиваться как минимум до
этого уровня (`trendStart`, §8).

**Это усиление, а не ранний гейт.** Способ измерения непринципиален
(прямая дистанция по барам или пивот старшего зигзага — §8); важно гарантировать
факт тренда достаточной длины. Поскольку порог зависит от волны B,
R-TREND окончательно проверяется **после подтверждения B** (фаза `WAIT_C`
и далее), до ARMED.

### 5.2. Почему тренд нужен даже после подтверждения B

Running-свойство (B за точкой 0) само по себе не гарантирует, что мы не в волне X:
то, что мы приняли за **C**, может оказаться волной **W** следующей структуры.
Единственная защита — требовать реальный тренд *до* точки 0. Поэтому R-TREND
проверяется на этапе точки 0 и не снимается при появлении B/C.

### 5.3. Containment волны A — тест «линии назад» (R-A-INIT)

Волна A обязана быть **начальной коррекцией внутри тренда**, а не улетать за его
начало. Геометрически: проводим горизонтальные лучи назад (в прошлое) от **конца
волны A** и от **точки 0**; двигаясь по свечам предшествующего тренда, луч от
конца A должен встретить тренд **раньше**, чем луч от точки 0. Если первым тренд
встречает луч от точки 0 — треугольник невалиден (A вышла за пределы тренда).

Это **тот же приём**, которым определяется «заходность» импульса
([`IsInitialMovement`](TradeKit.Core/Common/SingleSetupFinder.cs)). Более того,
тест буквально совпадает с вызовом:

```csharp
// leg 0 -> A трактуется как «импульс»; он должен быть initial внутри тренда
bool aContained = IsInitialMovement(
    startValue: V(point0),
    endValue:   V(waveA),
    startIndex: point0.BarIndex,
    bp:         BarsProvider,
    out _);
```

Развёрнуто для `isUp` (точка 0 — high, A — вниз), перематывая бары от точки 0:

```
for (i = point0.BarIndex - 1; i >= 0; i--) {
    if (High[i] >= V(point0)) return false; // тренд не «свежий» → A вышла за начало тренда
    if (Low[i]  <= V(waveA))  return true;  // уровень конца A достигнут внутри тренда
}
```

> **Важно для соотнесения.** Именно этот гейт (`IsInitialMovement(point0→A)`)
> ранее был **удалён**, т.к. убивал ~80% *всех* треугольников (для сжимающихся
> волна A корректирует необязательно тренд). Здесь мы возвращаем его **осознанно**:
> для running-треугольника A *обязана* быть начальной коррекцией тренда, поэтому
> тест уместен и является профильным фильтром, а не шумом.

## 6. Сигнал (entry / TP / SL)

Когда собран валидный кандидат и подтверждена волна E:

- **Триггер E**: E — последний закрытый сегмент зигзага, `V(A) < V(E) < V(D)`,
  и откат E составляет **≥ 50% амплитуды D**: `|V(D) − V(E)| ≥ 0.5·|V(D) − V(C)|`.
  Требование половины D нужно ради лучшего risk-reward.
- **Вход**: сразу по подтверждению E (market), как решено ранее для триангл-детектора.
- **Take-Profit — настраиваемый, по умолчанию конец волны B** (running → тяга
  уходит дальше точки 0), с `PERCENT_ALLOWANCE_TP` и округлением до
  `Symbol.Digits` (как в `ImpulseSetupFinder`). Альтернатива (флаг
  `TakeProfitMode`) — более консервативный TP на `point0`.
- **Stop-Loss = за волной A**, с `PERCENT_ALLOWANCE_SL` и округлением.
- Отбрасываем, если TP/SL уже задет на баре входа или если
  `|level − TP| / |TP − SL| < MIN_TO_SL_RATIO` (слишком близко к SL).

### 6.1. Повторные сигналы при росте треугольника вбок (настраиваемо)

Если мы **уже выдали сигнал**, а треугольник продолжает расти вбок
(перестроение §7.3), он может давать **новые сигналы** — например, когда
обновился экстремум новой волны C, а затем снова собрались D и E.

Поведение управляется флагом `EmitRebuildSignals`:

- `false` (**по умолчанию**) — один сигнал на треугольник (по точке 0);
  после ARMED перестроения больше не выдают сигналов.
- `true` — каждое новое подтверждение E после очередного роста вбок выдаёт
  новый сигнал (та же точка 0, обновлённые C/D/E → новые TP/SL).

> При `EmitRebuildSignals=true` дедупликация по точке 0 (§7.2) **не**
> блокирует повторные сигналы того же кандидата — она лишь не даёт создать
> второго кандидата на ту же точку 0.

## 7. Инкрементальный алгоритм (жизненный цикл кандидата)

Основная идея: **экстремумы зигзага в прошлом не меняются**, поэтому кандидатов
надо не пересобирать каждый бар, а **хранить и мутировать** до нарушения правил.

### 7.1. Состояние кандидата

```
RunningTriangleCandidate {
    BarPoint Trend0;      // точка 0 (конец тренда, начало треугольника)
    BarPoint A, B, C, D;  // подтверждённые вершины (могут быть null по фазе)
    Phase phase;          // WAIT_A | WAIT_B | WAIT_C | WAIT_D | WAIT_E | ARMED
    double trendStart;    // уровень начала тренда (для отмены salvage, §7.5)
}
```

Живёт **набор** кандидатов (разные точки 0 порождают разные кандидаты).

### 7.2. Обновление по новому сегменту зигзага

На каждой закрытой свече обновляем базовый `DeviationExtremumFinder`. Когда
появляется **новый подтверждённый экстремум** (новый сегмент зигзага):

1. **Порождение**: каждый новый экстремум, которому предшествует направленное
   движение — кандидат в **точку 0**; создаём кандидата в фазе `WAIT_A`.
   **Дедупликация по точке 0**: на один и тот же экстремум-точку 0 держим
   не более одного кандидата (повторные сигналы того же кандидата при
   росте вбок — это §6.1, не новый кандидат). Порог R-TREND (≥ |волна B|)
   окончательно проверяется позже (когда B известна, §5.1).
2. **Продвижение фазы**: новый сегмент пытается стать очередной волной текущей
   фазы кандидата, если проходит соответствующий инвариант §4:
   - `WAIT_A` → приходит A (коррекция); проверяем R-A-INIT → `WAIT_B`.
   - `WAIT_B` → приходит B; проверяем R-B-RUN → `WAIT_C`.
   - `WAIT_C` → приходит C; проверяем R-C-0 → `WAIT_D`.
   - `WAIT_D` → приходит D; проверяем R-D-B → `WAIT_E`.
   - `WAIT_E` → приходит E; проверяем R-E (+ R-CONTAIN всех волн) → `ARMED`
     → выдаём сигнал (§6).

### 7.3. Перестроение «треугольник растёт вбок»

Типичная ситуация: волна E уходит **за волну C, но не пробивает волну A**. Тогда
треугольник просто «раздаётся вбок»:

- **оставляем `{0, A, B}`**, отбрасываем `C, D, E`;
- считаем, что весь участок бывших C…E — это **новая волна C**, кандидат
  переходит в фазу `WAIT_D`;
- ждём новых сегментов зигзага для D и E.

Формально: в `WAIT_E`, если `V(E) < V(C)` (для `isUp`) и `V(E) > V(A)` (A цела),
то `C ← <новый экстремум>`, `D ← null`, `phase ← WAIT_D`.

Если кандидат **уже выдавал сигнал** (был ARMED) и при флаге
`EmitRebuildSignals=true`, то после повторной сборки D/E из этой же фазы
выдаётся новый сигнал (§6.1). При `false` перестроение продолжается
(кандидат живёт), но новых сигналов он больше не даёт.

### 7.4. Инвалидация

Если **пробита волна A или B** до того, как кандидат стал `ARMED`:

- по умолчанию — **кандидат целиком невалиден** (тренд, точка 0, ABCDE),
  удаляем его и начинаем заново;
- **исключение при пробитии именно A** — см. salvage (§7.5).

Для `isUp`: «пробита B» = `High > V(B)` (и это не новая валидная D…), «пробита A»
= `Low < V(A)`.

### 7.5. Salvage при пробитии волны A

При пробитии волны A допускаем альтернативу: бывшая **B становится точкой 0**
нового треугольника, а всё движение трактуется как **волна A нового** running-
треугольника (новый кандидат в фазе `WAIT_B`).

Этот сценарий **отменяется**, если движение в ту же сторону продолжается и
пробивает **точку начала тренда** (`trendStart`). Способ проверки — см. §8.

## 8. Точка начала тренда (`trendStart`) и отмена salvage

`trendStart` — уровень, до которого должен дотягиваться тренд перед точкой 0
(R-TREND, §5.1), и пробитие которого отменяет salvage (§7.5).

**Способ измерения непринципиален** (решение по O-1) — важен лишь факт
тренда длиной ≥ |волна B|. Берём простой вариант:

```
trendStart = V(0) ∓ |V(B) − V(A)|        // против тяги: − для isUp, + для isDown
```

Тренд считается достаточным, если цена до точки 0 реально доходила до
`trendStart` (проверка перематкой баров назад от точки 0, как в R-A-INIT).
Опционально вместо дистанции можно взять реальный пивот старшего зигзага
до точки 0 — результат эквивалентен, на выбор реализации.

> **Отмена salvage.** После пробития волны A (§7.5) новый кандидат
> (бывшая B → новая точка 0) отменяется, если движение в ту же сторону
> пробивает `trendStart` исходного тренда.

## 9. Предлагаемая архитектура

- **Новый класс** `RunningTriangleSetupFinder : SingleSetupFinder<ElliottWaveSignalEventArgs>`
  в `TradeKit.Core/ElliottWave/`.
- **Состояние**: `List<RunningTriangleCandidate>` + один базовый
  `DeviationExtremumFinder` (ABCDE) + опционально старший (тренд/salvage, §8).
- **`CheckSetup(openDateTime)`**:
  1. Отработать активный сетап (TP/SL-хиты) — как сейчас в `TriangleSetupFinder`.

     кандидатам, породить новых.
  3. При переходе кандидата в `ARMED` — сформировать сигнал (§6), `OnEnterInvoke`.
- **Отказ от общего markup**: `TryValidateTriangleV2` / `CheckPriceRules` в этом

- `Diag`/`Bump` из текущего детектора переносим (funnel-диагностика гейтов).

## 11. Решённые вопросы

- **O-1. Тренд / `trendStart`.** Тренд до точки 0 — это **усиление**, способ
  измерения непринципиален. Достаточно гарантировать тренд длиной **≥ длины
  волны B** (§5.1, §8; см. скриншот). Старший зигзаг не обязателен.
- **O-2. Дедупликация — по точке 0** (§7.2), с учётом повторных сигналов при
  росте вбок (§6.1).
- **O-3. Отдельный `RunningTriangleSetupFinder`** (§9). Текущий
  `TriangleSetupFinder` остаётся как есть.
- **O-4. TP — настраиваемый, по умолчанию конец волны B** (§6).
- **O-5. Полная зеркальность.** Один код обслуживает бычьи и медвежьи
  сетапы через `isUp`; «high/low» в правилах не фиксируются, всё через
  «по/против тяги» (примеры с high/low в доке — только иллюстративны).
